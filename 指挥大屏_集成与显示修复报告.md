# 指挥大屏：集成与显示修复报告

> 生成日期：2026-07-22 ｜ 范围：`/command-screen` 指挥大屏
> 结论先行：**两个问题均已通过真实浏览器（puppeteer + 既有 Chrome）实测定位并验证修复，未做任何猜测性改动。**

---

## 一、真实根因

### 1. 显示问题（内容被裁切）—— CSS Grid 列宽撑爆
- `CommandScreen.vue` 的 `.cs-root` 是 `display: grid`，**只显式声明了 `grid-template-rows`，未声明 `grid-template-columns`**。
- 缺少显式列定义时，网格会生成**一条隐式 `auto` 列**。在 CSS Grid 中，`auto` 列宽取「所有网格项 min-content 的最大值」。
- 而 `.cs-root` 的各级子元素（`.cs-row-header` / `.cs-row-kpi` / `.cs-main` / `.cs-row-ticker`）**都没有 `min-width: 0`**，于是该列被内部内容（KPI 5 列栅格 + 底部跑马灯 min-content）撑到 **~2501px**（在 1440×900 视口下，视口内容宽仅 1418px）。
- 外层 `.fullscreen-layout { overflow: hidden }` 把超出 **~1099px** 的右侧整体裁掉 —— 表现就是**顶栏时钟/状态、第 5 张 KPI 卡片被切掉**，且因为是被 `overflow:hidden` 裁切而非滚动，所以**页面不会出现横向滚动条**（用户描述的「异常横向滚动条」未出现，但「内容被裁切」确凿）。
- 这是纯前端布局缺陷，**与后端数据契约、数据量、窗口缩放（zoom）均无关**。

### 2. 集成问题（像独立页，不像系统内模块）
- `/command-screen` 路由 `meta.layout: 'fullscreen'`，由 `FullscreenLayout` 承载，正确与 `AppLayout`（常规侧边栏布局）隔离。
- 但 `AppLayout.vue` 的侧边栏导航**没有任何指向 `/command-screen` 的入口**，用户只能手动输入 URL 进入 —— 因此它表现为「独立展示页」而非「系统内的一个模块」。
- 同时屏内**缺少返回常规系统的出口**，进入后无法便捷回到正常业务流。

## 二、实际修改清单（均在允许范围内）

| # | 文件 | 改动 | 性质 |
|---|------|------|------|
| 1 | `frontend/src/views/CommandScreen.vue` `<style scoped>` | `.cs-root` 增加 `grid-template-columns: minmax(0, 1fr)`；`.cs-row-header/.cs-row-kpi/.cs-main/.cs-row-ticker` 增加 `min-width: 0` | 显示修复（核心） |
| 2 | `frontend/src/layouts/FullscreenLayout.vue` | `onMounted/onBeforeUnmount` 将 `document.documentElement.style.zoom` 置空（防御性；`theme.css` 当前已为 `zoom:1.0`，无需再强制）。`.fullscreen-layout` 的 `inset:0; width:100vw; height:100vh; overflow:hidden` **未改动** | 清理冗余 reset |
| 3 | `frontend/src/components/AppLayout.vue` | 侧边栏新增「指挥大屏」入口 `→ /command-screen`（带 `nav-item--screen` 视觉强调）；`pageTitle`/`pageSub` 增加 `/command-screen` 映射 | 集成（方案 A） |
| 4 | `frontend/src/components/command-screen/ScreenHeader.vue` | 右上角新增「↩ 返回系统」按钮，点击 `emit('exit')`；新增 `.cs-exit` 样式 | 集成（屏内出口） |
| 5 | `frontend/src/views/CommandScreen.vue` `<script>` | 引入 `useRouter`，新增 `exitScreen()` → `router.push('/dashboard')`，并在 `<ScreenHeader>` 上绑定 `@exit="exitScreen"` | 集成（出口逻辑） |

**未触碰**：后端数据契约 / 数据库 / WebSocket·SSE / 大型依赖 / 旧 `Dashboard.vue` 逻辑 / Phase 1 数据口径 / `App.vue` 的布局分发逻辑（本来就已正确）。

## 三、修复前 / 后 运行时尺寸与溢出（真实浏览器实测）

实测视口 1440×900（视口内容区 `clientWidth=1418, clientHeight=802`）：

| 指标 | 修复前（bug） | 修复后 |
|------|--------------|--------|
| `.cs-main / .cs-header / .cs-kpi-bar / .cs-row-ticker` 宽度 | **2501px** | **1386px** |
| 上述元素 `right`（视口内容宽 1418） | **2517**（超出 1099px，被裁切） | **1402**（≤ 1418，完整可见） |
| 顶栏时钟 `.cs-clock` `right` | **2511（已在屏外，被裁切）** | **1402（可见）** |
| `document.documentElement.scrollWidth / clientWidth` | 1418 / 1418 | 1418 / 1418 |
| `document.documentElement.scrollHeight / clientHeight` | 802 / 802 | 802 / 802 |
| 横向滚动条 | 无（被 `overflow:hidden` 裁切） | 无（内容完整，无需裁切） |
| 主要内容是否被裁切 | **是（右侧 ~1099px 被切）** | **否** |

**五档尺寸全量复核**（1280×720 / 1366×768 / 1440×900 / 1536×864 / 1920×1080）：
- 修复后各尺寸下 `.cs-root` 的 `right` 均 **== clientWidth**（贴合视口），`header/kpi/ticker` 的 `right` 均 **≤ clientWidth`。
- 五档均 `scrollWidth == clientWidth && scrollHeight == clientHeight`，**无文档级横向滚动条、无内容右侧裁切**。
- 唯一剩余的「溢出元素」是屏内信息流列表（`.cs-feed-list` 内若干 `.cs-feed-row`）其 `bottom` 超过视口但 `right` 不超 —— 它们位于 `overflow:hidden` 的面板内（预期的滚动信息流），被面板裁切而非文档裁切，**不影响整体布局与滚动条**。

## 八、遗留说明 / 建议（非本次范围，供后续决策）

1. **屏内信息流纵向溢出**：屏内「实时快讯 / 预警滚动」列表在极矮视口下内容多于面板高度，由面板 `overflow:hidden` 裁切（预期滚动流）。如需完整可见，后续可评估面板内独立滚动，但属体验增强，不在本次「修复裁切」范围内。
2. **`static/assets` 累积旧哈希**：`_d.py` 只写不删 + Vite `emptyOutDir:false`，旧分块会累积；建议后续在 CI/部署中加入清理步骤（沙箱批量删除受限，未本次处理）。
3. **axios 无全局 401 拦截器**：依赖路由守卫；旧 token 失效时屏会静默空屏。此前已在记忆中记录建议（遇 401 跳 `/login`），待拍板后实施。
