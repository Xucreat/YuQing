# 数据源配置模型与采集器装配可靠性优化 — 实施报告

> 执行日期：2026-07-23　范围：舆情监测系统（FastAPI 后端 + Vue3 前端 + PostgreSQL 16）
> 方法论：**先审计 → 后实施 → 再验证 → 最终报告**；最小化修改，不过度抽象。

---

## 1. 目标与约束回顾

| 目标 | 状态 |
|---|---|
| 防止专用型采集器被误填通用 `config_json` | ✅ 已实现（后端 422 + 前端隐藏输入） |
| 从后端阻止无效配置进入数据库 | ✅ create + update 两端均已拦截 |
| 前端明确区分"专用型 / 通用型"采集器 | ✅ 列表徽章 + 配置弹窗差异化 |
| 改善采集器装配失败时的错误可见性 | ✅ 装配失败写入 `CollectorRun(status=failed)` |
| 不破坏已有 9 个专用型数据源及其采集逻辑 | ✅ 9 个专用类源码零改动（git 确认） |
| 不进行过度抽象 | ✅ 仅新增 1 个校验函数族 + 1 个 verbose 装配入口 |

**关键校验原则（严格遵守）**：未知配置字段 **明确报错，绝不静默忽略**。例如 `{"keywrods": ["河北"]}` 会被明确拒绝（422），而非被悄悄丢弃。

## 2. 第一阶段：只读审计结果

连接生产库（`opinion_db` @ 127.0.0.1:5432，真实库 `舆情监测系统\pgdata`）只读核查：

- **数据源总数：30**（其中当前后端 API 视图返回 20 条启用源）。
- **`config_json` 存储现状**：9 条存 `"{}"`（专用型）、21 条存合法非空 JSON（通用型）、**0 条** null / 空串 / 非法 JSON。
- **结论：无需数据迁移**——现存 9 个专用源已全部为空配置，符合目标模型。

**审计发现的三大可靠性风险（本次全部修复）**：

1. **PATCH 端点校验缺口**：`update_data_source` 原先只做 JSON 可解析性检查，**未区分采集器类型**——可通过直接 API 调用把 `{"keywords": [...]}` 写入专用型数据源。
2. **静默配置降级**：`registry._parse_config` 遇到非法 JSON / 非对象时 **静默返回 `{}`**，掩盖配置错误，运行时表现为"配置好像没生效"。
3. **静默装配失败**：`resolve_collectors` 中 `except Exception: logger.error(...)` 后 **继续（相当于 pass）**，失败的数据源被无声地从采集队列剔除，操作者无从察觉。

## 3. 目标配置模型（实现后的语义）

```
              ┌──────────────────────────────┐
config_json → │ class_path == GenericSite... ?│
              └───────────────┬──────────────┘
              是（通用型）      │      否（专用型）
        ┌───────────────┘      └───────────────┐
        ▼                                       ▼
 必须非空 + 字段白名单校验              config_json 必须为空（{}）
 未知字段 → 422 明确报错              非空 → 422（DEDICATED_EMPTY_HINT）
        │                                       │
        ▼                                       ▼
 由 config_json 驱动抓取                使用采集器自身内置逻辑

  任何错误配置 → 不允许静默进入系统 → 必须被明确拒绝或暴露
```

## 5. 后端行为变化

### 5.1 装配可靠性（`registry.py`）

- 新增异常 `ConfigParseError`。
- `_parse_config`：非法 JSON / 非对象 **抛 `ConfigParseError`**（不再静默 `{}`）；仅 `None`/`""`/`{}`/空 dict 归一化为空配置。
- 新增 `@dataclass ResolvedCollectors { collectors: List, failures: List[dict] }`。
- 核心逻辑抽为 `_resolve_core(db, collector_type)`：装配失败（`ConfigParseError` 或 `cls(**cfg)` 抛 `TypeError`）时 **记录到 `failures`**（含 `key/name/class_path/error`），而非静默跳过。
- **向后兼容**：`resolve_collectors()` 仍返回 `List[BaseCollector]`（测试/脚本依赖不变）；新增 `resolve_collectors_verbose()` 返回 `ResolvedCollectors`。

### 5.2 配置校验（`admin_data_sources.py`）

新增常量与辅助函数：

- `GENERIC_ALLOWED_KEYS`（10 个）：`source_name, list_urls, link_rule, content_selectors, keywords, max_articles, request_interval, timeout, max_retries, retry_backoff`。
- `GENERIC_LINK_RULE_KEYS`：`href_contains, href_regex, href_exclude, title_blacklist, max_links`。
- `DEDICATED_EMPTY_HINT = "当前采集器为专用型采集器，无需填写自定义配置。请保持配置为空（{}）。"`
- `_is_config_empty(raw)`：None / 空串 / `"{}"` / `{}` 均视为空；非法 JSON 不视为空（交后续报错）。
- `_is_generic(class_path)`、`_validate_generic_config(cfg)`（**未知字段明确报错，不静默过滤**；含 `link_rule` 子字段校验）。

**校验分支（create 与 update 一致，无覆盖缺口）**：

| 采集器类型 | 空配置 | 非空且字段合法 | 含未知字段 |
|---|---|---|---|
| 通用型 (Generic) | 422 拒绝（"必须提供 config_json"） | ✅ 通过 | 422 拒绝（列出允许字段） |
| 专用型 (Dedicated) | ✅ 通过（归一化为 `"{}"`） | 422 拒绝（DEDICATED_EMPTY_HINT） | 422 拒绝（DEDICATED_EMPTY_HINT） |

- `create_data_source`（POST）与 `create_and_test`：尾部调用 `_validate_create` 分支校验。
- `update_data_source`（PATCH）：`config_json` 处理块补齐同款校验——通用型拒绝 null/空/未知字段，专用型只允许 null/""/"{}"/{}（归一化为 `"{}"`），其余一律 422。
- `_serialize` 输出新增 `"collector_kind": "generic" | "dedicated"`。

### 5.3 装配失败可见（`service.py`）

- `collect_and_analyze`（定时/顺序）与 `collect_and_analyze_concurrent`（手动/并发）入口均：先生成 `run_start` + `batch_id`，用 `resolve_collectors_verbose()` 装配，对每条 `failures` 调用新方法 `_record_assembly_failure()`。
- `_record_assembly_failure()` 写入一条 `CollectorRun(collector_name=name, batch_id=batch_id, trigger_type=trigger_type, start_time=run_start, end_time=now, status="failed", error_msg=error)`。
- 效果：**装配失败的数据源不再无声消失**，会出现在采集日志（`collection-logs` 批次）中，`status=failed` + 具体 `error_msg`。
- `CollectorRun` 模型已具备 `batch_id / trigger_type / error_msg` 列（前序"采集日志"阶段引入），本次直接复用，**无新增迁移**。

## 7. 装配可靠性变化小结

| 场景 | 优化前 | 优化后 |
|---|---|---|
| 非法 JSON 配置 | 静默降级为 `{}` | 抛 `ConfigParseError`，装配失败记入 CollectorRun |
| 专用型误填 config | 可能被 `cls(**cfg)` 的 TypeError 静默丢弃 | 装配失败记入 CollectorRun（可见）；且后端根本不允许写入 |
| 某源装配异常 | `except: pass` 静默剔除 | `failures` 记录 → `CollectorRun(status=failed, error_msg)` |

## 12. 结论

本次优化在 **不破坏任何现有专用型数据源、不做过度抽象** 的前提下，从三条链路彻底堵住了"错误配置静默进入/生效"的隐患：

- **入口拦截**：后端 create + update 双端按采集器类型强校验，未知字段明确报错。
- **装配透明**：`_parse_config` 不再静默降级，装配失败写入 CollectorRun 可见。
- **人机提示**：前端明确区分专用/通用型，专用型直接隐藏配置输入。

所有验证（语法、API 6 场景、相关单测、前端构建/部署、最小运行验证）均达预期，改动范围严格限定在 4 个文件内。
