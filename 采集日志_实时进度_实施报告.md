> 实施日期：2026-07-23
> 实施立场：企业级功能增强，严格遵循「只读审计 → 小范围修改 → 构建 → 验证 → 部署」流程；不引入 SSE/WebSocket、不新增 API 路径、不改数据库结构/迁移、不重构采集链路、不扩大范围。

## 1. 总体结论

**已完成并验证通过。** 用户在「数据管理 → 采集日志」页面点击「开始采集」后，无需手动刷新即可实时看到本次采集任务的执行进度（批次、已运行时间、总数、已完成、运行中、成功/部分/失败、当前正在处理的数据源、后台阶段），采集完成后进度区自动收尾、历史日志自动刷新、新批次出现在列表中。

实施采用**前端 Polling 轮询**（复用既有 `pollTask` 基建 + `collection-logs?status=running`），后端仅做**最小关联改动**（`task_id ↔ batch_id`）。全程**未新增 API 路径、未改数据库结构、未做迁移、未新增依赖**。

一处超出预期但必要的后端小修：原 `collection-logs` 接口的 SQL 已聚合 `running_count` 但 `_to_item` 未将其输出（用户 §四.D 期望该字段存在），补上该字段（非破坏性，仅为既有响应增加键）。

## 4. task_id ↔ batch_id 关联实现方式

**方案（比原 Prompt 建议更优、最小侵入）：** 在 `_run_collect_task` 内**采集开始前**即生成 `batch_id` 并写入 `Task.batch_id`，再将其传入 `collect_and_analyze_concurrent(batch_id=...)`。服务内部 `batch_id = batch_id or uuid.uuid4().hex` 保证单一来源。

**为何优于原建议：** 原建议「服务返回结果携带 batch_id、任务结束后再写 task」会使 `task.batch_id` 在采集进行中一直为 `None`，前端首轮轮询拿不到 batch_id，无法在采集过程中定位批次。本方案让 `task.batch_id` 从**第一次轮询**即可用，真正实现「采集中实时进度」。

**可靠定位保证：** `GET /api/tasks/{task_id}` 经 `to_dict()` 返回 `batch_id`；前端拿到后按 `batch_id` 精确匹配 `collection-logs?status=running` 中的批次，再查 `/{batch_id}/runs`。即使手动采集与定时采集重叠，也能精确对应「用户点的那一次」。

---

## 5. 实时轮询实现方式

**复用既有基建，不引入新协议。**
- **Task 轮询**：复用 `api`（axios）对 `GET /api/tasks/{task_id}` 轮询，间隔 **1.5s**（递归 `setTimeout`，非 `setInterval`，避免请求重叠），获取 `status/progress/step/batch_id/error`。
- **批次进度轮询**：`GET /api/admin/data-sources/collection-logs?status=running`（2s）按 `batch_id` 精确匹配当前批次，取聚合统计；再 `GET /api/admin/data-sources/collection-logs/{batch_id}/runs` 取逐源实时状态。
- 两路轮询均带 `inFlight` 式守卫（`live` 标志 + 递归 setTimeout），卸载即停。

## 6. 当前采集任务 UI 实现方式

历史表格上方独立卡片「当前采集任务」（仅 `live=true` 时显示），与历史日志视觉/语义明确隔离：
- 标题「当前采集任务」+ 触发方式胶囊（手动=蓝 / 定时=灰）+ 批次 `#xxxxxxxx`。
- 核心进度：已运行时间（mm:ss，1s 计时）、数据源总数、已完成、运行中。
- 结果统计：成功（绿）/ 部分成功（橙）/ 失败（红）。
- 当前正在处理：真实 `status='running'` 的数据源名标签。
- 阶段信息：`task.step`（如「已采集 8/22 个数据源」「采集完成，正在自动聚合事件…」）。

## 7. 采集进度数据来源

- 总数 / 成功 / 部分 / 失败 / 运行中：来自 `collection-logs?status=running` 中匹配 `batch_id` 的批次聚合项（`source_count/success_count/partial_count/failed_count/running_count`）。
- 已运行时间：`batch.started_at` 与本地时钟差（时区与既有 `dur()` 一致，无 8h 偏移）。
- **不使用** `on_progress` 的 `name` 当「正在处理」——该 `name` 是刚完成源名（见 §8）。

---

## 8. 当前正在处理数据源的判断方式

**严格依据 DB 真实状态：** 对当前批次调 `GET /{batch_id}/runs`，筛选 `status === 'running'` 的 `collector_name` 列表展示。验证中实时看到如「大厂县政府网站 / 承德市政府网 / 百度新闻 / 河北新闻网 …」等真实在跑的源，随采集推进动态减少。**未使用** `on_progress(name)`（其为刚完成源，语义错配）。

---

## 9. 聚合阶段处理方式

以 **Task 状态为任务生命周期依据**：
- 采集源全部完成 → CollectorRun 全终态、但 `Task` 仍 `running`（正在 `auto_aggregate_after_collect`）。
- 此时 `running_count===0`，但批次可能已不在 `status=running` 列表 → 批次轮询判定为「聚合中」，保留上次计数并停批次轮询，进度区继续显示（阶段信息显示 `task.step`「采集完成，正在自动聚合事件…」）。
- 仅当 `Task` 进入 `success`/`failed` 才 `finishLive()`。验证中 Task 最终 `status=success`、`step=采集与自动聚合完成`，进度区随之收尾。

---

## 10. 页面刷新恢复方式

- `activeTaskId`/`activeBatchId` 持久化于 localStorage。`CollectionLog.vue` `onMounted` 时若 `collectStore.activeTaskId` 存在 → `startLive(activeTaskId)` 恢复轮询（先查 task 拿 batch_id，再轮询批次）。
- 额外 `watch(activeTaskId)`：用户在**同页**点击「开始采集」时即时启动，无需刷新。
- 无 `activeTaskId` 时不自动展示（**保守处理**，避免把历史 running 孤儿误判为当前任务）。
- 已知 `task_id` 但后端重启致 `task` 404：若已持 `activeBatchId` → 切换为按 `batch_id` 继续轮询数据库进度；若无 batchId → 安全收尾。

---

## 11. 异常处理方式

- **采集触发失败（CollectorThrottled / 5 秒防抖）**：`task.status==='failed'` 且 error 含 `frequently`/`CollectorThrottled` → `ElMessage.warning('采集操作过于频繁，请稍后再试。')`；其他失败 → `ElMessage.error('采集失败：…')`，随后 `finishLive`。
- **Task 查询失败/404**：按 §10 降级到 batch 轮询或安全收尾，不永久卡「正在采集」。
- **网络瞬断**：轮询递归重试，不无限叠加（单轮询链唯一）。
- **401**：既有 axios 拦截器统一登出跳转；轮询捕获 401 即 `stopLive()`，不产生无限请求。
- **组件卸载**：`onUnmounted(stopLive)` 清空全部定时器。

---

## 12. 定时器清理方式

三路定时器（`taskTimer`/`batchTimer` 用 `setTimeout` 递归，`elapsedTimer` 用 `setInterval`）统一由 `clearLiveTimers()` 在 `stopLive()` 中清除；`onUnmounted` 调 `stopLive`。进入→离开→再进入页面不会叠加多个轮询器（每次 `startLive` 先 `stopLive` 再启）。
