# 事件中心重复事件 · 根因分析与解决方案（只读排查）

> 范围：仅检查，不修改任何代码/数据。目标：解释"事件中心存在两条完全相同的事件（如『生态遇上非遗 湿地公园"潮"起来等2条相关舆情聚集』）"的成因，并给出修复方案。

## 1. 现象与量化（生产库 opinion_db，只读 SELECT）

| 指标 | 数值 |
| --- | --- |
| events 总数 | 237 |
| 相同 title 的重复事件组（dup≥2） | **43 组** |
| 冗余事件数（多余副本） | **43 个** |
| 典型样本 | id 336 与 339 标题均为"2026年交通运输产教融合系列活动在廊坊开幕等2条相关舆情聚集"，**关联舆情集合完全相同 `{392, 1091}`** |

- 重复模式高度一致：**几乎每个聚合事件都成对出现**（title 完全相同，关联舆情集合相同），是系统性而非偶发。
- 用户所举"生态遇上非遗 湿地公园"潮"起来等2条相关舆情聚集"即属此类（title 含"等2条相关舆情聚集"，由 `build_cluster_title` 生成，代表舆情相同+成员数相同 → 两次物化标题必然相同）。

## 2. 根因分析

### 2.1 直接机制：聚合器物化无并发保护，出现竞态重复创建

聚合触发链（`app/services/event/aggregator.py`）：

1. 采集后自动聚合：`auto_aggregate_after_collect()`（L682）→ `EventAggregator().aggregate(db, incremental=True)`（L697）。
2. 增量路径 `_aggregate_incremental()`（L360）：
   - 召回"窗口内 + completed + **未关联任何事件**"的舆情（L376-385，`Opinion.id NOT IN (SELECT opinion_id FROM event_opinions)`）。
   - 聚类后物化为新事件 `_create_event()`（L315 全量路径 / 增量对应分支）。
3. 防重复只靠 `_match_existing_event()`（L514）：
   - **共享成员匹配**（L523-532）：簇中任一 Opinion 已挂到某事件 → 沿用。
   - **事件延续**（L534-550）：无共享成员时，按 representative 的 `_merge_condition` + 时间接近匹配活跃事件。

**竞态窗口**：候选召回（L376）与"是否已关联"判定（L523）都依赖**已提交事务**的可见性，而两次聚合之间**没有任何锁/事务隔离/串行化**。

### 2.2 触发条件：Phase 7 之前的多实例双触发

- 历史运行中存在**两个后端实例**（:8000 与 :8011）各自启动调度器，每整半点采集被触发 **2 次**（间隔约 55ms）。
- 每次采集后都调用 `auto_aggregate_after_collect`，于是**同一批舆情在 ~55ms 内被并发聚合两次**：
  - 实例 A 召回未关联候选 (O1,O2) → 聚成簇 → 物化 EventA，flush 关联但**尚未 commit**；
  - 实例 B 在 A commit 前召回，同样读到未关联 (O1,O2) → 同样物化 EventB；
  - 两者均 commit → **EventA 与 EventB 完全相同（同标题、同关联舆情）**。
- Phase 7 已通过 advisory lock 实现调度单例（每整半点仅 1 批），**根因触发条件已消除**，但**历史残留的 43 个冗余事件仍存在**。

### 2.3 结构缺陷（为何 Phase 7 后仍有可能再生）

- 聚合器**无跨进程/跨请求串行化锁**：即使单实例下 asyncio 单线程协作式调度使真正并发概率极低，但只要出现"手动聚合 `/api/events/aggregate` 与采集后自动聚合叠加"或"未来扩多实例"，竞态即复发。
- **无事件级幂等约束/去重键**：events 表只有 `title`/`keyword` 文本字段，无 (规范化标题 + 时间桶) 或 representative opinion 的唯一约束兜底；防重完全依赖软匹配逻辑，弱于 DB 约束。

### 2.4 与"keyword 重复"区分（避免混淆）

`events.keyword` 存在大量共用通用词（"群体"20、"事故"13、"投诉"12…），这是**不同舆情共享内置 16 敏感词**导致的"低区分度小事件过多"问题（聚合噪声/过聚维度），**不属于本次"完全相同的重复事件"**，应作为独立优化项单列，不在本次修复范围。

## 3. 解决方案

分两层：**A. 历史去重（修已存在）+ B. 根因修复（防再生）**。均建议遵循 Phase 7 已确立的运维纪律（事务内执行 + 删除前 SELECT 验证 + pg_dump 备份 + 不自动删业务数据）。

### A. 历史去重（43 个冗余事件）

1. 生成重复组：按 `title` 完全相同分组，对每组取两事件关联舆情集合，**集合相同/高度重叠**即判定为冗余副本。
2. 合并规则（事务内）：
   - 保留一个（建议 `min(id)` 或 `last_time` 较新者），将另一副本的 `event_opinions` 迁移到保留事件（`INSERT ... SELECT ... WHERE NOT EXISTS`，受 p7 唯一约束 `uq_event_opinions_event_opinion` 保护自动去重）；
   - 处理 `propagation_nodes.event_id` 引用（先 `UPDATE ... SET event_id=保留id WHERE event_id=副本id`，或置空/重建传播树）；
   - 删除冗余事件行；
   - 对保留事件 `rebuild_for_event` 重建传播树 + 重算派生字段（opinion_count/last_time/risk_level/keyword/title）。
3. 验证（COMMIT 前 SELECT）：同 title 事件组=1、冗余事件=0、`propagation_nodes` 无悬空 event_id、传播树节点数合理。
4. 纪律：先 `pg_dump` 备份；脚本参考 `backend/cleanup_duplicate_opinions.sql` 的事务化写法。

### B. 根因修复（防再生，按优先级）

**B1（高，最小改动，推荐先做）— 聚合串行化锁**
- 在 `EventAggregator.aggregate()` 入口用 **PostgreSQL 会话级咨询锁**（`pg_try_advisory_lock`，key 由 `sha1("event-platform-aggregate-singleton")` 派生），与 `app/core/scheduler.py` 的 `_try_acquire_scheduler_lock()` 同款模式。
- 仅在持有锁时执行聚合，未持有则跳过（或排队）。从根上消除跨进程/跨请求并发竞态，**无需改表结构、不引入 Redis**。

**B2（中，纵深防御）— 事件级幂等去重键/约束**
- 方案 a：聚合创建事件前，先用"representative opinion 的归一化特征 + 时间窗口"查活跃事件（`_match_existing_event` 已存在，强化为"首选按代表舆情 id/标题指纹匹配"）。
- 方案 b（更稳）：events 表新增 `dedup_key`（如 `md5(规范化标题)` 或 representative opinion_id），对"活跃窗口内"加部分唯一索引；或在 `_create_event` 前 `INSERT ... ON CONFLICT` 兜底。需新增 Alembic 迁移（参考 p6/p7 写法）。

**B3（低，部署层）— 单实例部署**
- Phase 7 已建议关停 :8011 冗余实例；单实例下 asyncio 协作式调度使聚合竞态概率趋近于零，与 B1 叠加后彻底消除。

### 优先级建议
1. **B1（加聚合锁）**：1 处改动、立即防再生，与现有 scheduler 锁同构，风险极低 → 建议作为下一阶段第一个修复。
2. **A（历史去重）**：清理 43 个冗余事件，恢复事件中心质量 → 紧随 B1 之后执行（先锁后清，避免清理过程中又产生新重复）。
3. **B2（去重键/约束）**：长期纵深防御，可在后续迭代实施。
4. **keyword 过聚**治理（2.4）：独立优化项，不在本次范围。

## 4. 结论

- **用户所见"两条完全相同事件"是真实存在的系统性重复**：生产库 237 个事件中 **43 个为冗余副本**，由 Phase 7 之前的多实例双触发 + 聚合无并发串行化触发，**同一簇被物化两次**。
- **Phase 7 已消除触发条件（调度单例）**，但历史冗余残留且聚合器本身仍缺锁/约束，**仍有理论再生可能**。
- 本次为**只读排查**，未修改任何代码或数据。建议下一阶段按 B1→A→B2 顺序修复。
