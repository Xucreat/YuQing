# 预警中心展示优化方案（Phase 7.x-C 前置审计 + 设计）

> 阶段性质：只读审计 + 设计方案。**未修改任何代码与数据**。
> 前置依赖：风险评分治理（opinions 重算）+ 历史误判预警处置（31 条 false_positive 已标记）均已落地。

---

## 一、当前问题

1. **误报记录仍默认可见**
   - 历史治理已将 31 条"原 critical→low、severity=0"记录标记为 `status=false_positive`、`handled=True`。
   - 但预警中心列表接口 `GET /api/alerts/records` **没有"排除某状态"的能力**，前端也没有对应开关。
   - 结果：预警中心默认把这些已归位误报的记录与真实风险混在一起展示，用户仍需手动翻找，达不到"默认可信"的治理目标。

2. **筛选维度不全、体验割裂**
   - 后端仅支持 `risk_level`（等级）与 `handled`（是否已处理布尔）两个过滤参数。
   - 前端提供了"预警等级""处理状态"两个下拉，但**没有"处置状态(status)"筛选**——而 `status` 才是 5 态企业级处置流（pending/processing/resolved/ignored/false_positive）的权威字段。
   - `handled` 布尔与 `status` 语义重复且易混淆（handled 仅区分"是否终态"，无法区分 resolved/ignored/false_positive）。
   - 用户无法按"误报/已忽略/处理中"精确筛选，定位有效预警效率低。

## 四、设计方案

### 目标
- **默认隐藏 false_positive**，让预警中心首屏只展示可信风险；提供开关可随时查看误报（保留审计）。
- **补齐 status 筛选**，支持按 5 态处置流精确过滤，替代含义模糊的 `handled` 布尔筛选。

### 4.1 后端最小改动（`backend/app/api/alerts.py` → `list_records`）

新增两个可选 Query 参数，仅在传入时追加 `where`，不影响既有无参调用：

```python
@alerts_router.get("/records", response_model=AlertRecordListResponse)
def list_records(
    page: int = Query(1, ge=1),
    size: int = Query(20, ge=1, le=MAX_SIZE),
    risk_level: str | None = None,
    handled: bool | None = None,
    status: str | None = None,            # 新增：按处置状态过滤
    exclude_status: str | None = None,    # 新增：排除某状态（如 false_positive）
    db: Session = Depends(get_db),
    _u: User = Depends(get_current_user),
):
    q = db.query(AlertRecord)
    if risk_level:
        q = q.where(AlertRecord.risk_level == risk_level)
    if handled is not None:
        q = q.where(AlertRecord.handled == handled)
    if status:                                      # 新增
        q = q.where(AlertRecord.status == status)
    if exclude_status:                              # 新增
        q = q.where(AlertRecord.status != exclude_status)
    total = q.count()
    rows = q.order_by(AlertRecord.id.desc()).offset((page - 1) * size).limit(size).all()
    return AlertRecordListResponse(items=rows, total=total, page=page, size=size)
```

**参数语义与边界**
- `status`：取值应为 5 态之一；建议加 allow-list 校验（复用 `_ALLOWED_HANDLE_STATUS` 集合），非法值返回 422。
- `exclude_status`：`status` 列有 `NOT NULL + CheckConstraint`，无 NULL 干扰，`!=` 排除干净；传入 `'false_positive'` 即隐藏全部误报。
- 两者可组合：`status` 精确命中 + `exclude_status` 排除，逻辑为 AND（用户同时选 status=false_positive 且开隐藏开关 → 空集，属预期，无需特殊处理）。
- **默认值**：两参数默认 `None` ⇒ 不改变既有调用行为，向后兼容。

**关于"默认隐藏"落点（两种可选）**
- **推荐（前端默认开关 ON）**：后端保持 `exclude_status` 默认 `None`；前端"隐藏误报"开关默认开启，默认发 `exclude_status=false_positive`。好处：后端零行为变更、其它潜在调用方不受影响。
- 备选（后端级默认隐藏）：将 `exclude_status` 默认值设为 `'false_positive'`。好处：即使直连 API 也默认隐藏；坏处：改变所有调用方默认表现，需评估 dashboard/unread 等是否受影响（本方案不改动它们，故不采用）。

### 4.2 前端交互设计（`frontend/src/views/Alerts.vue`）

1. **新增"处置状态"下拉**（替换/补充现有 `recFilterHandled`）
   - `recFilterStatus`：el-select，clearable，选项 pending/processing/resolved/ignored/false_positive，placeholder "处置状态"，`@change="loadRecords"`。
   - 建议**保留** `recFilterHandled`（向后兼容旧接口参数），但 UI 文案明确"处理状态(终态/未终态)"，避免与 status 混淆；或仅保留 status 下拉、隐藏 handled 下拉（handled 仍可作底层兼容参数保留发送）。

2. **新增"隐藏误报"开关**
   - `hideFalsePositive`：el-switch，默认 `true`（即默认隐藏）。
   - 关闭时发 `exclude_status` 为空 ⇒ 展示全部（含误报）。

3. **`loadRecords()` 参数补齐**（第 222–225 行）：
   ```ts
   const params: any = { page: recordsPage.value, size: recordsSize.value }
   if (recFilterRisk.value) params.risk_level = recFilterRisk.value
   if (recFilterHandled.value !== null) params.handled = recFilterHandled.value
   if (recFilterStatus.value) params.status = recFilterStatus.value          // 新增
   if (hideFalsePositive.value) params.exclude_status = 'false_positive'     // 新增（默认开）
   const { data } = await api.get<AlertRecordListResponse>('/alerts/records', { params })
   ```

4. **新增 ref 声明**：`const recFilterStatus = ref('')` / `const hideFalsePositive = ref(true)`。

5. **状态列已具备**：现有 `statusText/statusTag`（第 77、201–207 行）可直接展示 5 态，无需改动；误报记录若被展示，其标签为 `info` 灰色"误报"，与真实风险视觉区分清晰。

| 文件 | 改动 | 性质 |
|---|---|---|
| `backend/app/api/alerts.py` | `list_records` 增 `status`、`exclude_status` 两 Query 参数 + 2 条 where 条件（可选、向后兼容） | 后端只读过滤 |
| `frontend/src/views/Alerts.vue` | 增"处置状态"下拉 + "隐藏误报"开关；`loadRecords()` 补发 `status`/`exclude_status`；增 2 个 ref | 前端 UI/交互 |

**不需要修改**：`alert_service.py`、`risk_engine.py`、采集链路、任何模型/迁移、数据库结构、`AlertRecordOut` 响应结构（status 字段已存在）。
