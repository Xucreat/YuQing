## 一、业务语义验收

### 四层语义是否清晰

| 语义层 | 现状 | 结论 |
|---|---|---|
| 1. 数据源管理 = 管理单个数据源 | 独立 tab，表格一行一个源，行内"查看历史/配置"按钮 | ✅ 清晰 |
| 2. 单数据源历史 = 某个源自己的历史 | 弹窗（dialog）形态，标题"采集历史 · {源名}"，明确绑定源名 | ✅ 清晰 |
| 3. 采集日志 = 一次采集批次的整体执行 | 独立 tab"采集日志"，副标语境"共 N 次采集批次"，一行一批 | ✅ 基本清晰 |
| 4. 批次详情 = 该批中每个源的执行结果 | 行内展开（inline expand），逐源表格 | ✅ 可用，但见【三】的弱点 |

### 混淆风险评估：单源历史 vs 全局批次日志

**混淆风险：低。** 理由（有意或无意做对了三件事）：
1. **形态不同**：单源历史是弹窗，批次日志是页面 tab——模态差异天然区隔两个概念；
2. **入口不同**：一个从数据源行进入，一个从顶层 tab 进入；
3. **标题锚定**：弹窗标题带源名（"采集历史 · 邯郸市政府网"），不会被误读为全局日志。

**残余风险（轻微）**：批次详情展开后的逐源表格与单源历史弹窗**列结构和视觉样式高度相似**（复用 hist-tbl 样式），用户在批次详情中看到"邯郸市政府网 一行"时，语义是"该源在本批次的一次执行"，而弹窗里同样的行语义是"该源的第 N 次历史执行"。因两者不会同时出现在屏幕上，判定为**暂不处理**。

## 二、采集日志列表验收

### 逐项检查

| 检查项 | 结论 | 说明 |
|---|---|---|
| 信息过多 | **⚠️ 是** | 12 列。成功/部分/失败/抓取/新增/分析 6 个纯数字列并排，当前数据下 partial/failed 恒为 0，"0 0 0"噪声明显 |
| 横向过宽 | **⚠️ 临界** | 见响应式推算（§六）：1366×768 下处于挤压临界区，1440+ 无碍 |
| 重要信息不突出 | **⚠️ 是** | "状态"排第 11 列（倒数第 2），是用户最先想看的字段却在最右侧；失败数与成功数视觉权重相同 |
| 状态不易识别 | **❌ 是（最严重）** | `running` 映射为**绿色药丸**，与 `success` 同色系；45% 的行显示"运行中"且为绿色，用户无法区分"正常"与"未完成" |
| 时间不易理解 | ✅ 否 | `2026-07-23 15:09:16` 格式规范，精确到秒 |
| 触发方式不清晰 | **⚠️ 部分** | 手动=蓝/定时=灰可辨；但 125/128 的历史批次显示灰色"未知"，占绝对多数，业务用户会困惑"未知是什么" |
| 成功/部分/失败统计不直观 | **⚠️ 是** | 三个独立数字列，无颜色、无占比感；"22 0 0" 不如 "22/22 成功" 直观 |

### 字段分级（仅评估，不修改）

**【核心字段】**（一屏必看，决定用户是否要点进去）
- 采集时间（started_at）
- 触发方式（手动/定时/历史）
- 整体状态（success/partial/failed/…）
- 数据源数 + 成功数（建议合并表达为 "22/22"）
- 失败数（仅当 >0 时才有信息量，应醒目）

**【次要字段】**（有价值但可弱化/合并）
- 耗时（可与时间列合并或缩窄）
- 新增（业务上"这次采到了几条新数据"其实较重要，介于核心与次要之间）
- 部分成功数（当前恒为 0）

**【可以进入详情查看的字段】**
- 抓取（fetched_raw）—— 原始抓取量是技术指标
- 分析（analyzed）—— 与新增强相关，明细里逐源看更有意义
- 逐源开始/结束时间、错误信息（已在详情，正确）

> 若按此分级收敛，主表可从 12 列减到 7~8 列，1366 宽度问题随之消解。

## 三、批次详情验收

| 检查点 | 结论 | 说明 |
|---|---|---|
| 1. 明确"这是一次采集批次" | ⚠️ 部分 | 行内展开保留了父行上下文（时间/触发/状态就在上一行），无独立标题栏声明"批次"。展开区内**没有任何批次级摘要头** |
| 2. 明确显示批次时间 | ✅（借父行） | 展开区自身不显示，滚动后父行仍可见（父行不 sticky，长明细滚动时上下文会滚出视野——明细区 max-height 50vh 内部滚动，父行实际不会滚走，可接受） |
| 3. 明确显示触发方式 | ✅（借父行） | 同上 |
| 4. 明确显示整体状态 | ✅（借父行） | 同上 |
| 5. 明确显示数据源总数 | ⚠️ 弱 | 父行有"数据源数"列，但展开区无"共 22 个源"摘要；明细分页 size=50，若一批 >50 源会**静默截断**（当前 22 源，不触发） |
| 6. 快速定位失败/部分成功的源 | **⚠️ 弱** | 无排序（失败未置顶）、无状态筛选；靠红色药丸肉眼扫描。22 行尚可接受，源数增长后低效 |
| 7. 数据源明细是否易读 | ✅ 是 | 列齐全（源名/开始/结束/耗时/抓取/新增/分析/状态/错误），nowrap+sticky 表头+内部滚动，错误红色。唯一小瑕疵：`err-cell max-width:320px` 与 `nowrap` 并存，超长错误文本会溢出单元格（无 ellipsis） |
| 8. 与单源历史产生视觉/语义混淆 | ✅ 基本无 | 见§一分析，样式复用带来一致性收益大于混淆风险 |

## 四、历史数据验收

- **batch_id=NULL 展示**：后端以 `COALESCE(batch_id, start_time::text)` 兜底分组，前端对用户完全透明（用户只看到时间），✅ 处理正确，无需 UI 变更。
- **trigger_type=NULL 展示**：前端显示灰色药丸"未知"。**评估：不适合业务用户。**"未知"暗示系统出了问题；实际语义是"该批次产生于功能上线前，未记录触发方式"。建议 UI 层面改为"**历史**"或"未记录"（灰色药丸，hover 提示"该批次产生于采集日志功能上线前"）。纯前端文案改动，一行即可。
- **历史 running 批次**：**这是全页面最大的误导源**（详见§五）。57 个孤儿批次全部显示绿色"运行中"，实际没有一个在跑。
- 结论：**不需要修历史数据，只需要 UI 表达层优化**（与用户约束一致）。

## 五、状态语义验收（重点）

### 四态在批次级的用户理解匹配度

| 状态 | 显示 | 颜色 | 用户理解匹配 |
|---|---|---|---|
| success | 成功 | 绿 | ✅ 匹配 |
| partial | 部分成功 | 橙 | ✅ 匹配（当前无数据验证） |
| failed | 失败 | 红 | ✅ 匹配 |
| running | 运行中 | **绿** | ❌ **双重错位** |

### running 的双重错位（判定：必须修复，但均为前端最小改动）

1. **颜色错位**：`runPill` 将 `running → pill-green`，与 success 同绿。运行中语义应为蓝色（进行中）而非绿色（完成且良好）。
2. **语义错位**：57/128 批次是数据库真实状态为 running 的**孤儿记录**（进程中断/崩溃遗留，`end_time=NULL` 永不更新）。它们同时显示"运行中 + 耗时 81.3s"——自相矛盾（还在跑怎么会有耗时？）。业务用户会得出"系统有 57 个任务卡死在跑"的错误结论。

### 评估与最小方案（不改后端状态逻辑——未发现严重业务错误，孤儿是既有事实而非本次实现引入）

**纯前端展示层判定**（一个函数即可）：
```
若 status === 'running' 且 (批次 started_at 距今 > 2 小时)
  → 显示为「未完成」灰色药丸，hover 提示“该批次存在未正常结束的记录，可能因服务中断产生”
否则 → 显示为「运行中」蓝色药丸
```
- 不改数据库、不改后端、不改 API；
- 真实在跑的批次（2 小时内）仍正确显示"运行中"；
- 57 个孤儿全部变为诚实的"未完成"。
- 状态筛选器中的"运行中"选项因后端语义仍是 running，可保留（筛出的即"运行中+未完成"合集），或改标签为"运行中/未完成"。

## 六、响应式与布局验收

> ⚠️ 方法说明：本机无无头浏览器，以下为**布局参数精确推算**，非截图实测。参数来源：侧边栏固定 246px、内容区 `max-width:1440px; padding:34px 44px`、批次表 12 列（固定宽合计 840px，每单元格水平 padding 36px，时间列自适应），`.card` 为 `overflow:hidden`。

| 分辨率 | 表格可用宽 | 推算结论 |
|---|---|---|
| 1366×768 | ≈1030px | **临界**。12 列最小内容宽（含 padding）估算 945–1130px。大概率表现为：表头两字列换行、时间列换行折行、整体拥挤；**最坏情形下右侧"操作"列被裁切且无横向滚动条**（`.card overflow:hidden` 溢出即裁切，不出滚动条——这是隐患点） |
| 1440×900 | ≈1106px | 基本完整，略拥挤，操作列可见 |
| 1920×1080 | ≈1352px（受 max-width:1440 限制） | 完整舒适 |
| 批次详情展开区 | 内部 `overflow:auto` | ✅ 明细表 nowrap 但有自身横向滚动，任何分辨率不丢内容 |
| 分页 | el-pagination prev/pager/next | ✅ 正常，右对齐，total>size 才显示 |
| 筛选器 | 触发方式+状态两个下拉+刷新 | ✅ 布局合理（flex-wrap 可折行）。未暴露日期范围筛选（后端已支持 from/to），当前数据量下不必需 |

**布局风险根因**：不是屏幕不够宽，而是**列太多**（§二的字段分级已给出收敛路径）。另一保险做法：`.card` 的 `overflow:hidden` 改 `overflow-x:auto`（一行 CSS），保证任何情况下内容可达。

## 七、总体结论

### 1. 是否可以直接投入使用

**可以投入使用（有条件）**：功能正确性、数据准确性、历史兼容、权限门禁、空/加载/错误态均验收通过；四层业务语义清晰不混淆。
**条件**：running 状态的表达问题（颜色 + 孤儿语义）建议在正式面向业务用户前修复——45% 的行呈现误导性状态，这不是"UI 完美主义"，而是会直接产生错误业务认知（"57 个任务在跑/卡死"）。

### 2.【必须修复】（共 2 项，均为前端小改，合计 <20 行代码）

| # | 问题 | 最小改动方案 |
|---|---|---|
| M1 | running 绿色 + 57 个历史孤儿显示"运行中"，产生"大量任务在跑"的错误认知 | `CollectionLog.vue` 展示层：`runPill` 中 running 改蓝色；增加超龄判定（started_at 距今 >2h 且 status=running → 显示"未完成"灰药丸 + title 提示）。不改后端、不改数据 |
| M2 | `.card overflow:hidden` 在 1366×768 下溢出即裁切"操作"列，无滚动条可达 | `.card` 改 `overflow-x:auto`（1 行 CSS）。彻底兜底，任何分辨率内容可达 |

### 3.【建议优化】（4 项，可合并为一次小迭代）

| # | 问题 | 建议 |
|---|---|---|
| S1 | 12 列过多、状态列偏居末位、"0 0 0"噪声 | 按§二字段分级收敛为 7~8 列：合并"成功/部分/失败"为"成功率 22/22"（失败>0 时红色标注失败数）；"抓取/分析"移入详情；状态列前移至触发方式旁 |
| S2 | 触发方式"未知"对业务用户不友好 | 文案改"历史"（或"未记录"）+ hover 说明，一行改动 |
| S3 | 批次详情无法快速定位失败源 | 明细请求后前端排序：failed/partial/running 置顶（纯前端 sort，一行） |
| S4 | 耗时 >60s 仍显示"136.7s" | 格式化为"2分17秒"样式 |

### 4.【暂不处理】（明确不建议现在做）

| # | 事项 | 理由 |
|---|---|---|
| N1 | 批次详情展开区加批次摘要头 | 行内展开已保留父行上下文，收益低 |
| N2 | 明细 >50 源静默截断 | 当前 22 源，且源数量增长缓慢；备注在案即可 |
| N3 | 日期范围筛选 UI | 后端已支持，128 批次量级下翻页足够；数据量上千后再加 |
| N4 | 单源历史与批次明细样式趋同 | 一致性收益大于混淆风险（两者不同屏出现） |
| N5 | 历史数据回填 batch_id | COALESCE 兜底工作正常，回填属数据修改，违背最小干预 |
| N6 | CollectionLog 懒加载分包 | 与全站现状一致，属全局优化议题 |
| N7 | err-cell 超长错误文本溢出 | 出现频率低，详情区可横向滚动兜底；若做 S1 迭代可顺手加 ellipsis |

### 5. 最小改动方案汇总

若只做【必须修复】：改动仅限 `frontend/src/views/CollectionLog.vue` 单文件（runPill/runText 增加约 10 行判定逻辑 + 1 行 CSS），vite build + `_d.py` 重新部署即可，**零后端改动、零数据库改动、零 API 变更、可即时回滚**（保留有旧构建备份目录）。
