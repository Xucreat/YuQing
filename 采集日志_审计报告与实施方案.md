## 4. CollectorRun 生命周期

【已验证事实】
- **唯一创建点**：`service.py:253`（位于 `_process_collector` 内部）。全代码库 `CollectorRun(` 仅出现两处：模型定义 + 此创建点。
- **每次采集触发都会为每个数据源各建一条 `CollectorRun`**（遍历 `self.collectors`）。
- **同一次触发的所有 `CollectorRun` 共享同一个 `run_start`**（`run_start` 在 `collect_and_analyze` / `collect_and_analyze_concurrent` 顶部生成一次，原样传入每个 `_process_collector`）。
- **失败 / 异常 / 重试**：
  - 单条 Opinion 分析失败：被 `try/except` 隔离，`opinion.analysis_status="failed"`，但 `CollectorRun` 仍正常写回 `status`（`success` 或 `partial`）。**不会**新建 `CollectorRun`。
  - 整个 `_process_collector` 抛异常（如 `collector.fetch` 崩溃未捕获）：`CollectorRun` 已先 `commit`（status=`running`）后崩溃 → 该 run 永远停在 `running`。这是历史库中 `running` 状态的来源（见 §5/§7）。
  - **没有重试机制**：代码无 retry/覆盖逻辑；一个数据源在一次触发中**至多一条** `CollectorRun`（DB 实证：duplicate (start_time, collector_name)=NONE）。
- **不会出现**：一个 `CollectorRun` 对应多个采集动作（`CollectorRun` 与一次 `_process_collector` 调用一一对应）；重试覆盖原记录（无重试）。
- **部分数据源没有 `CollectorRun`**：仅当该源未进入 `_process_collector`（如 `resolve_collectors` 装配阶段异常/被禁用）。批次仍按 `start_time` 聚合，只是 `source_count` 小于启用源总数。

【已验证事实】`duplicate (start_time, collector_name)` 配对 = **NONE** → 同一批次内不存在"同源多条"。

## 7. 历史数据兼容性分析（核心）

【已验证事实】
- 历史 1791 行按 `start_time` 聚出 **126 个批次**，每批内部 `start_time` 完全一致、无同源重复、end_time 跨度合理 → **`start_time` 能可靠代表一次采集批次**。
- 微秒精度（precision 6）+ 一次触发只生成一次 `datetime.now()` → 不同触发碰撞概率极低；生产数据中间隔均为分钟级（13:59 / 13:55 / 13:51…），**实证无碰撞**。
- 【存在不确定性】理论上两触发在**同一微秒**生成 `start_time` 仍可能（概率≈0，且数据无实例）；属理论边界，非实际风险。
- 【已验证事实】历史行 `trigger_type` 不可恢复（列不存在）。`start_time` 分钟落在 `:00`/`:30` 且秒≈0 的多为 cron 批次，其余为手动——可用启发式近似，但属推断，不强行回填。

【建议方案】
- **可以直接按 `start_time` 兼容历史数据**，采用 `COALESCE(batch_id, start_time::text)` 作为批次键：
  - 新数据（迁移后）：`batch_id` 非空 → 按 `batch_id` 聚合。
  - 历史数据（`batch_id` 为 NULL）：自动回退按 `start_time::text` 聚合。
- **历史 `trigger_type` 保持 NULL**，前端显示为"未知（历史数据）"；筛选按触发方式仅对新数据生效（不强行回填，符合"历史数据不要求强行回填"）。
- 不采用"强行 `COALESCE` 但 start_time 不可靠"的反模式——本系统 start_time 经实证可靠，故方案成立。

## 9. 原方案哪些判断需要修正 / 补强

【基于代码的判断】
1. **原方案把 `run_start` 当作"隐式批次键"直接 GROUP BY**，未充分考虑：历史 `trigger_type` 不可恢复，且 `running` 状态的孤儿 run 会影响批次状态判定。→ 修正：批次状态需显式按子 run 状态汇总（见 §10/§14），`running` 孤儿须纳入"未完成/失败"语义。
2. **原方案未指出 `status` 中存在 62 条 `running` 孤儿**，批次状态聚合若简单"全部 success=成功"会误判。→ 修正：批次状态优先级 `running > failed > partial > success`。
3. **原方案未说明批次详情接口如何兼容两种键**（UUID vs start_time 字符串）。→ 修正：列表返回 `batch_key = COALESCE(batch_id, start_time::text)`，详情接口用 `WHERE COALESCE(batch_id, start_time::text) = :batch_key` 统一解析。
4. **trigger_type 对历史为 NULL**：筛选按触发方式对历史无效，需在前端明确标注，不回填。

## 10. 最终推荐实现方案

【建议方案 · 经审计确认适配当前架构】
- **模型**：`CollectorRun` 增加 `batch_id`（String(64)，索引）、`trigger_type`（String(16)，可空）。
- **批次边界（真正触发边界生成 batch_id）**：
  - `collect_and_analyze`（定时入口）→ `trigger_type="scheduled"`，方法顶部生成一次 `batch_id`。
  - `collect_and_analyze_concurrent`（手动入口）→ `trigger_type="manual"`，方法顶部生成一次 `batch_id`。
  - `batch_id` 在 fan-out **之前**生成，作为只读参数传入 `_process_collector`（**不在** `_process_collector` 内部重新生成）。并发路径中 `batch_id` 为不可变字符串，跨线程安全。
- 一次性脚本/测试沿用 `collect_and_analyze` 默认值 `trigger_type="scheduled"`（不影响生产数据）。
- **不新建** `collection_batches` 表（用户已确认优先在 `CollectorRun` 上加列）。
- **历史兼容**：`COALESCE(batch_id, start_time::text)` 聚合，无需回填。

## 11. 不推荐的方案及原因

【建议方案】
- ❌ 仅按 `start_time` 分组、不加 `batch_id`：能跑，但依赖"同触发共用 run_start"的实现约定，未来改动易崩；且无法区分手动/定时。不推荐。
- ❌ 独立 `collection_batches` 表 + 外键：过度设计，用户已明确不要求；增加迁移与写库复杂度。
- ❌ 强行启发式回填历史 `trigger_type`：属推断、可能误标，违反"不强行回填"。
- ❌ 在 `_process_collector` 内生成 `batch_id`：会导致"一个数据源一个 batch_id"，破坏同批次共享。

## 14. API 设计

【建议方案】
- `GET /api/admin/data-sources/collection-logs`
  - 查询参数：`page`(1)、`size`(20)、`trigger_type`(manual|scheduled，可选)、`status`(可选)、`from`/`to`（ISO，可选）。
  - 聚合 SQL：`GROUP BY COALESCE(batch_id, start_time::text) AS batch_key`，取 `MIN(start_time) started_at`、`MAX(end_time) finished_at`、`COUNT(*) source_count`、各计数求和、`success/partial/failed/running` 计数（CASE WHEN）、`MAX(trigger_type) trigger_type`、`SUM(fetched_raw/created/analyzed)`。
  - 数据量小（当前 126 批），全量聚合后在 Python 侧做筛选（trigger_type/status/日期）与分页，逻辑更清晰。
  - 批次状态（基于真实状态体系，不发明新状态）：
    `running_count>0 → "running"`；`elif failed_count>0 → "failed"`；`elif partial_count>0 → "partial"`；`else "success"`。
  - 返回：`items[]`（batch_key, batch_id|null, trigger_type|null, started_at, finished_at, duration_seconds, source_count, success_count, partial_count, failed_count, fetched_raw, created, analyzed, status）、`total`、`page`、`size`。
- `GET /api/admin/data-sources/collection-logs/{batch_key}/runs`
  - `WHERE COALESCE(batch_id, start_time::text) = :batch_key`，`order_by(start_time.desc(), id)`。
  - **复用**现有 `/{id}/runs` 的逐行序列化（id, collector_name, start_time, end_time, fetched_raw, created, analyzed, failed, status, error_msg）；额外可附 `duration_seconds`。
  - 分页 `page/size`。

## 16. 风险清单

【存在不确定性 / 风险】
1. **历史 `running` 孤儿（62 条）**：会使对应批次状态判定为 `running`（未完成）。对旧批次显示为"运行中/未完成"是诚实的；若需严格"失败"语义可后续调整，本期按 `running` 优先。
2. **`start_time` 微秒碰撞（理论）**：概率≈0 且无实例，采用 `COALESCE(batch_id, start_time::text)` 后新数据完全规避。
3. **历史 `trigger_type=NULL`**：按触发方式筛选对历史无效，前端标注"未知（历史数据）"。
4. **迁移对象为生产库**：已通过身份门禁；加列为非破坏性、可 downgrade。
5. **并发 `batch_id` 线程安全**：在 fan-out 前生成、只读传入，无共享可变状态。
6. **环境无无头浏览器**：前端验收靠 `vite build` 成功 + 运行时拉取 + 逻辑核对（既有验收习惯）。
