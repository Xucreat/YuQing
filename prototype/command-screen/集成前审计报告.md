## 2. 原型核对（任务②）

- **组件依赖**：ECharts 5.5.1（CDN）、DataV 中国 GeoJSON（CDN）、Google Fonts（Rajdhani / Share Tech Mono / Noto Sans SC）；纯原生 DOM，无框架。

- **数据字段**：`total/today/high_risk/event_count/trend/keywords/sources/sentiments/regions/recent/alerts` 与后端 dashboard 端点一一对应，mock 结构与真实 schema 字段名一致。
- **WebSocket 预留**：`connectWS()` 骨架（注释态），按 `type:stats|recent|alert` 增量推送；`boot()` 末尾注释掉。
- **直接复制到 Vue 的兼容性问题**：

  7. 原型 3s 全量轮询直连无缓存后端 → 直接搬会打爆后端（见 E1）。

## 4. Opinion.keywords 语义专项（任务④）

- **真实语义**：`collectors/service.py:258` `opinion.keywords = ",".join(analysis.keywords)`；`analysis.keywords` 来自 `RuleFallbackProvider.analyze`（`ai/fallback.py:75-77,127`）——遍历**内置敏感词表 `DEFAULT_KEYWORDS`**，把"出现在文本中的敏感词"作为 keywords。即：**敏感词命中集合**，不是监测词表关键词，也不是 AI 抽取主题。
- **实测**：429 条中 362 条（84%）为空；非空样本 = `事故 / 群体 / 腐败 / 死亡`（典型敏感词）。
- **是否适合做"热门关键词"**：❌ **不适合**。
  - 覆盖率极低（仅 16% 有值），大屏会显得稀疏；
  - 语义是"命中了哪些敏感词"，不是"当前在讨论什么话题"，作为"热门"会误导（高频的永远是 民生/投诉/事故 这类固定词）；
  - 与"监测关键词表"（keywords 表）是两回事，也不是 `ai_keywords`（AI 抽取，仅手动触发才有、更稀疏）。
- **更合适的数据源**：
  - 监测关键词表 `keywords`（`keyword_service.get_monitoring_keywords`）——是固定配置列表，**无频次**，语义是"系统在什么词上监测"，宜作"监测词覆盖"展示，不宜作"热门"。
  - `ai_keywords`——语义最贴合"话题"，但仅手动触发 AI 分析才有值，覆盖极低且依赖 DeepSeek。
  - **结论**：当前系统没有现成的"按频次的热门话题"数据源。
- **建议**：新增独立聚合接口 `/dashboard/hot-keywords?days=N`，对窗口内 opinions 做主题词频统计（推荐：基于监测关键词表统计各词在窗口内命中/提及频次，或中文分词对 title+content 抽取名词短语排名）。**无论如何不要把 `Opinion.keywords` 当大屏热门词。**

## 5. 地图数据专项（任务⑤）

- **当前 regions 是否支持省级 choropleth**：❌ 不支持（直接）。
  - regions 表 17 条：province 1（河北省）+ city 12 + county 4（固安/香河/大厂/三河），高度河北化。
  - `opinions.region_id` 分布 province 170 / city 75 / county 184 → 聚合出的 `region_name` 同时含"河北省 / 廊坊市 / 大厂回族自治县"。
- **地域名称是否统一**：❌ 不统一，省/市/县混用。原型用 `stripSuffix` 去后缀匹配 DataV 省级 GeoJSON，但 `100000_full.json` 只有省级 feature，市/县名匹配不到 → **地图大面积空白**。
- **是否已有 GeoJSON 资源**：❌ 项目内无任何本地中国地图 GeoJSON/TopoJSON（仅原型运行时从 DataV CDN 拉取）；内网大屏若无外网则地图加载失败。
- **是否需要地域标准化**：✅ 需要。
  - **方案 A（推荐，后端改）**：`dashboard_service` 新增"按省上卷"——用 `Region.code` 前缀（前 2 位=省、前 4 位=市）把 opinions 上卷到省级再输出 `province_name + count`。注意 `大厂回族自治县(131028)` 的 `parent_code=None`，**前缀法（1310→廊坊市 131000→河北省 130000）比 parent_code 更稳**。
  - 方案 B（前端改）：前端拿到混级 regions 后用"市/县→省"映射表上卷——映射维护成本高、且有边界问题，不推荐。
- **涟漪热点坐标**：原型用硬编码省会经纬度；真实数据上卷到省后，可用省份 centroid 静态表或 GeoJSON 计算质心。

## 6. 实时刷新方案对比（任务⑥）

| 维度 | 轮询（优化版） | SSE | WebSocket |
|------|----------------|-----|-----------|
| 数据更新频率 | 采集定时任务每 **30 分钟**批量到达，非高频流 | 同左，服务端有新批次即推 | 同左 |
| 客户端数量 | 大屏通常 **1–3 块**，单组织内网 | 同左 | 同左（WS 在多客户端才显优势） |
| 是否需双向 | ❌ 纯展示，无需 | ❌ 单向推送 | ✅ 支持双向（本场景用不上） |
| 后端改造 | **零**（端点已有）+ 加缓存 | 中（一个流式端点） | **高**（连接管理器+事件总线+与采集解耦+鉴权） |
| 大屏部署 | 最稳，穿透任何代理 | 需关 `proxy_buffering`、设 `text/event-stream` | 需 `Upgrade` 代理 |
| 断线重连 | 天然 | 浏览器 `EventSource` 原生重连 | 需自写重连（或库） |
| 数据一致性 | 有 10–15s 延迟，但底层数据 30min 才变，无感 | 近实时 | 近实时 |

**结论与建议**：
- **首选：优化轮询（带服务端 10–15s 缓存）**。理由：数据本身 30min 才变、客户端少、零新基建、最稳；把轮询间隔设为 15–30s（大屏对 30min 级更新无需 3s），后端给 `stats` 加短 TTL 缓存即可把 DB 压力降到可忽略。
- **次选：SSE**（若坚持"推送感"或未来要秒级）：实现成本低于 WS，浏览器原生重连，单端点即可。
- **WebSocket：当前不推荐作为默认**。它带来最大后端改造，而本场景无双向需求、更新低频、客户端少——WS 的优势（双向/低延迟/多路复用）用不上，反而增加连接管理/鉴权/重连复杂度。**仅当后续大屏需要"屏端下发控制指令/多人协作/海量客户端"时才上 WS**（那时原型里预留的 `connectWS()` 骨架可直接用）。
- **明确：不要默认 WS 一定最佳。**

## 7. 最终报告（A–H）

### A. 原型可直接复用部分
- 后端 `/dashboard/stats`、`/dashboard/recent`、`/dashboard/alerts` 三端点字段已对齐（仅需小修：加 `days` 统一窗口、regions 上卷省级、keywords 换源）。
- 前端现有鉴权链路：`api/index.ts` Bearer 注入、`useAuthStore`、`usePermission` composable（大屏路由加 `requiresAuth` 即可）。
- `Dashboard.vue` 的 ECharts option 可作**结构蓝本**（trend line / source bar / wordcloud），但需改暗色主题色、并把 region bar 替换为 map。
- 原型 README 中的**设计系统（深空蓝青令牌、布局、组件规范）**可直接作为大屏视觉基准。
- 原型预留的 `connectWS()` 骨架（未来上 WS 时复用）。

### B. 必须修改部分
- **[后端]** `regions` 上卷省级（dashboard_service），使地图拿到干净省级数据。
- **[后端]** 新增"热门关键词"独立聚合接口（不要复用 `Opinion.keywords`）；或先用监测词表频次兜底。
- **[后端]** `stats` 加 `days` 参数，统一收敛 `keywords/sources/sentiments/regions` 的时间窗口。
- **[后端]** `stats` 加服务端缓存（10–15s TTL，进程内/Redis），支撑轮询。
- **[前端]** 新增全屏布局（`FullscreenLayout` / `meta.fullscreen`），脱离 `AppLayout` 侧边栏 + `max-width` + `zoom`。
- **[前端]** 独立暗色主题作用域（深空蓝青 CSS 变量），不污染亮色 `theme.css`。
- **[前端]** ECharts 改 npm 依赖 + 按需引入，抽 `useEcharts` composable；地图 GeoJSON **vendoring 到 `public/`**（去外网依赖）。
- **[前端]** 数字 count-up / feed 滚动改为 Vue 组件或 composable。
- **[前端]** 补 401 响应拦截（token 过期跳登录），否则大屏常驻会持续报错。

### C. 数据语义风险
- **C1** `Opinion.keywords` = 敏感词命中（84% 空），≠ 热门关键词 → 必须换数据源。
- **C2** `regions` 省/市/县混用 → 地图匹配失败。
- **C3** `today`/`trend` 用 `created_at`（入库日）≠ `publish_time`（发布日）；大屏应标注"采集/入库"口径。
- **C4** `keywords/sources/sentiments/regions` 全量无窗口，与 `trend/today` 窗口不一致 → 全局态口径混乱。
- **C5** 情感为规则研判（非 AI），大屏应标注"系统研判"；negative 仅 3.5%，分布偏中性。
- **C6** `source` 自由文本，存在"政府网站"泛化 vs 具名站轻微碎片，榜单可能碎裂。
- **C7** 预警处置率仅 ~9%（1/11），大屏若展示"处置率"会是极低值，需决定呈现方式。

### D. 前端架构风险
- **D1** 布局不兼容全屏（侧边栏 + 居中 + zoom）。
- **D2** ECharts 全量引入无封装、无 ResizeObserver。
- **D3** 无响应拦截（401）、无 WS/SSE 基建。
- **D4** 仅单 auth store，无 theme/dashboard store。
- **D5** 路由不校验角色（若大屏需限制角色，要扩 meta + 守卫）。
- **D6** 现有 `Dashboard.vue` 与大屏功能重叠但视觉/布局不同，建议抽共享 chart composable 避免逻辑分叉。

### E. 后端接口风险
- **E1** 端点无缓存、全量重算（含 O(N) `keywords` Counter 扫描）；高频轮询下 DB 压力大。
- **E2** `regions` 未上卷省级，地图不可用。
- **E3** `keywords` 语义错配（见 C1）。
- **E4** 时间窗口不统一（见 C4）。
- **E5** `recent`/`alerts` 仅取最新 8 条、按 id/created_at desc，无分页/游标；大屏滚动要做"增量"需游标或推送。
- **E6** 端点均需鉴权；大屏常驻需保证 token 不过期，当前 JWT `expire=1440min`（24h）基本够，但无 refresh。

### F. 实时通信方案建议（详见 §6）
- **推荐**：优化轮询（间隔 15–30s + 后端 10–15s 缓存）。
- **备选**：SSE（要推送感 / 未来秒级时）。
- **不推荐默认 WS**；仅在有双向 / 高并发需求时启用（原型 `connectWS` 骨架可复用）。

### G. 推荐实施顺序
1. **后端小修（低风险高价值）**：`stats` 加 `days` 统一窗口 + `regions` 上卷省级 + 加 10–15s 缓存。先让"地图 + 全局态"拿到正确数据。
2. **新增热门关键词接口**（独立聚合），替换大屏关键词源。
3. **前端基建**：新增全屏布局 + 暗色主题作用域 + `useEcharts` composable（按需引入）+ GeoJSON vendoring。
4. **前端图表迁移**：把 `Dashboard.vue` 的 chart option 重构进大屏，套暗色令牌；实现 KPI / feed 组件。
5. **接数据**：用现有 `/dashboard/*` + Bearer；先轮询（15–30s）。
6. **上线前数据抽样验证**：跑附录 SQL 复核 regions 省级聚合、keywords 覆盖率、空值。
7. **（可选，二期）SSE 化**；仅当确实需要推送时再做。**本期不做 WebSocket 服务端实现。**

### H. 明确"暂时不要做"
- ❌ 不要把 `prototype/command-screen` 的 HTML 当 Vue 组件直接复制（CDN / 全量 echarts / 外网 GeoJSON / 手写动画都需改造）。
- ❌ **不要实现 WebSocket 服务端**（无双向 / 低频 / 少客户端，ROI 低；原型骨架留作备用）。
- ❌ **不要用 `Opinion.keywords` 作为大屏"热门关键词"**。
- ❌ 不要指望当前 `regions` 直接喂省级地图（必先做省级上卷）。
- ❌ 不要 3s 高频轮询直连无缓存端点（先加缓存 / 拉大间隔）。
- ❌ 不要改现有亮色 `theme.css` 去"适配"大屏（应独立暗色作用域）。
- ❌ 不要动 `AppLayout` 侧边栏去塞大屏（应独立全屏布局）。
- ❌ 在审计 / 数据修正完成前，**不要开始大屏 Vue 页开发**（避免基于错误数据建模）。

## 附录：上线前复测 SQL（只读）

```sql
-- 1) regions 上卷省级后的分布（验证地图数据）
SELECT LEFT(r.code,2) AS prov_prefix,
       CASE WHEN LEFT(r.code,2)='13' THEN '河北省' ELSE LEFT(r.code,2) END AS province,
       COUNT(o.id) AS opinion_cnt
FROM opinions o JOIN regions r ON r.id=o.region_id
GROUP BY 1,2 ORDER BY 3 DESC;

-- 2) keywords 覆盖率（验证是否仍大面积为空）
SELECT COUNT(*) FILTER (WHERE keywords IS NULL OR keywords='') AS empty_kw,
       COUNT(*) AS total FROM opinions;

-- 3) 空值核对
SELECT COUNT(*) FILTER (WHERE region_id IS NULL) AS null_region,
       COUNT(*) FILTER (WHERE sentiment IS NULL OR sentiment='') AS null_sent,
       COUNT(*) FILTER (WHERE risk_score IS NULL) AS null_risk
FROM opinions;

-- 4) 今日新增口径（入库日 vs 发布日差异）
SELECT COUNT(*) FILTER (WHERE created_at::date = CURRENT_DATE) AS by_created,
       COUNT(*) FILTER (WHERE publish_time::date = CURRENT_DATE) AS by_published
FROM opinions;

-- 5) 预警处置率
SELECT handled, COUNT(*) FROM alert_records GROUP BY handled;
```
