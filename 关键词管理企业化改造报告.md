# 关键词 / 敏感词管理企业化改造 — 最终报告

> 改造原则：**先审计，后决策；先复用，后扩展；先最小改造，后考虑未来演进。**
> 结论：**不预设"全部开放删除"，也不预设"全部硬编码不可改"**——采用
> **「系统内置词库 + 业务自定义词库 + 运行时启停/覆盖」** 的分层机制，
> 并复用既有 `keywords` 表（扩展字段），不新建重复 SensitiveWord 表。

---

## 一、现状审计（真实实现，以代码为准）

### A. 当前系统真实实现
1. **监测关键词（monitoring keywords）**：已有一套完整管理体系
   - 模型 `backend/app/models/keyword.py`（`keywords` 表：`id, word, weight, category`）
   - 服务 `backend/app/services/keyword_service.py`（`get_monitoring_keywords`，60s 缓存，表空回退 `settings.collector_keywords`）
   - API `backend/app/api/keywords.py`（CRUD + 分页 + 分类筛选，权限 `keywords:write`）
   - 前端 `frontend/src/views/Keywords.vue`（纯监测词管理 UI）
   - 用途：采集过滤 + 预警匹配（`alert_service.py` / `dashboard_service.py` 共用）
2. **敏感/风险词（sensitive words）**：**硬编码**，未入库
   - `backend/app/services/ai/fallback.py` 的 `DEFAULT_KEYWORDS`（16 词带权重，附分类）
   - 由 `RuleFallbackProvider` 在采集阶段（`collectors/service.py:270`）注入，生成 `risk_score`
   - **情感词表 `NEGATIVE/POSITIVE_SENTIMENT` 是独立硬编码表**，与风险词解耦
   - `aggregator.py` 也 import `DEFAULT_KEYWORDS` 用于 Event 信号分级（`GENERIC_KEYWORDS`）
3. **权限**：`permissions.py` 的 `require_permission("keywords:write")`（admin 直通），无独立读权限。
4. **迁移/种子**：`alembic` 线性链 HEAD=`ai0005`；`scripts/init_db.py` 播种 `MONITORING_KEYWORD_SEED`(29) 与 `DEFAULT_KEYWORDS`(16，仅作常量、未入库)。

### B. 已有能力
- 监测关键词完整 CRUD + 缓存即时失效 + 分类筛选；采集/预警/仪表盘消费。
- 风险评分由 `RuleFallbackProvider` 提供，离线可用、确定性强。

### C. 缺失能力
- **敏感词不可查看、不可管理、不可启停**——纯硬编码。
- 无法区分"系统内置"与"企业自定义"风险词。
- 无 `source/is_system/is_enabled/type/审计时间` 等分层字段。

### D. 可复用能力
- 既有 `keywords` 表、`Keyword` 模型、`keyword_service` 缓存模式、`keywords.py` API 骨架、前端 `Keywords.vue` 页面与样式、RBAC `require_permission`、采集注入点 `RuleFallbackProvider`（已支持外部 `keywords` 注入）。

### E. 不应重复建设的能力
- **不新建 SensitiveWord 表**（系统已有关键词管理体系，应扩展而非重复）。
- **不动 `aggregator.py` / `fallback.DEFAULT_KEYWORDS` / 情感词表**（保证 Event 聚合、风险评分、情感判定零回归）。
- 不引入多租户 / 审批流 / 版本管理 / 批量导入导出（任务未要求，当前架构无基础）。

### F. 建议的最小合理改造范围
扩展既有 `keywords` 表（加 `type`/`source`/`is_enabled`/时间戳 + 复合唯一 `(word,type)`），
把硬编码的 16 敏感词**幂等播种为 `type='sensitive', source='system'`**，
采集注入点改为"读库敏感词，空则回退 DEFAULT_KEYWORDS"，
API 增加分层过滤与安全守卫，前端页面增加分层展示/操作。

## 二、最终方案
**扩展既有 `keywords` 表，以 `type` 区分为两套词库：**
- `type='monitoring'`（监测关键词）：驱动采集过滤 + 预警匹配。业务可配置，管理员可正常增删改查与启停。
- `type='sensitive'`（敏感/风险词）：驱动风险评分（`RuleFallbackProvider`）。
  - `source='system'` 系统内置（init_db 播种的 16 词）：**可查看/搜索/筛选/启停，禁删禁改内容**。
  - `source='custom'` 管理员自定：可新增/编辑/删除/启停。

**为什么采用（契合当前架构）：**
- 系统已有关键词管理体系 → 直接扩展模型，零重复建设（符合"先复用后扩展"）。
- 敏感词与监测词业务语义不同 → 用 `type` 区分而非合并，概念清晰。
- 风险评分保留 `DEFAULT_KEYWORDS` 作为**兜底**：DB 无启用敏感词时行为与原版**逐字一致**，零回归。
- 安全边界用**数据层守卫**实现，不引入新权限体系（保留后续 RBAC 扩展空间）。

## 四、数据模型（系统内置 vs 自定义 如何区分）
同一张 `keywords` 表，用两字段区分：
- **`type`**：`monitoring`（监测词）/ `sensitive`（敏感/风险词）——业务语义分流。
- **`source`**：`system`（系统内置）/ `custom`（管理员自定义）——来源与受保护级别。
- **`is_enabled`**：运行时启停开关（不删数据即可下线某词）。
- 唯一性下沉为 **`(word, type)` 复合唯一**：监测词与敏感词可同名（如"舆情""投诉"），互不冲突。
- 审计：`created_at` / `updated_at`。

## 五、行为规则（明确）
| 操作 | 系统敏感词 (`source='system',type='sensitive'`) | 自定义词 (`source='custom'`) |
|------|------|------|
| 查看 / 搜索 / 按类型筛选 | ✅ | ✅ |
| 启停 (`is_enabled`) | ✅ 允许 | ✅ 允许 |
| 编辑内容 (word/weight/category) | ❌ **403 禁止** | ✅ 允许 |
| 删除 | ❌ **403 禁止** | ✅ 允许 |

- 监测关键词（`type='monitoring'`，无论 source）保持原有**全 CRUD + 启停**行为，不受"系统词禁删"规则影响（避免回归既有管理能力）。
- 删除/改内容的守卫在**数据层**实现（`_is_protected()`），当前复用 `keywords:write` 操作权限，保留后续接入 RBAC 的空间。

## 六、兼容性（对其他模块的影响）
- **采集**：`get_monitoring_keywords` 仅取 `type='monitoring' AND is_enabled=True` → 与改造前 29 词结果一致；采集过滤不受影响。
- **风险评分**：`RuleFallbackProvider` 现注入 DB 启用的 16 敏感词（= `DEFAULT_KEYWORDS`）→ **评分逐字不变**；DB 无启用敏感词时自动回退，离线/演示依旧可用。
- **分析 / 情感判定**：`NEGATIVE/POSITIVE_SENTIMENT` 未动 → 情感判定零回归。
- **告警**：`alert_service` 仍用监测词 → 不受影响。
- **Event 聚合 / Event-2 Narrative**：`aggregator.py` 与 `DEFAULT_KEYWORDS` 完全未改 → **不重建历史 Event、不重算 Narrative**。
- **指挥大屏 / 统计分析**：仅读取既有 API，未改契约。
- **缓存 / 定时任务**：沿用既有 60s TTL 缓存 + `clear_keyword_cache()` 即时失效，无新增定时任务。
