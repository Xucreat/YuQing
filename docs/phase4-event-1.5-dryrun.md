# Phase 4-Event-1.5 Event Rebuild Dry-Run & Data Migration Review

> 阶段性质：**只读验证**。本阶段对生产库 `opinion_db` 执行了一次**等价**于
> `EventAggregator.aggregate(rebuild=True, dry_run=True)` 的计算，但**全程仅 SELECT + 内存计算，
> 从未调用 `db.add` / `db.delete` / `db.commit`**。生产数据零改动。
> 依据严格禁令，本阶段**未执行任何正式 rebuild**。

## 1. Dry-Run 执行机制确认

读取 `backend/app/services/event/aggregator.py`（`aggregate()`）后确认：

| 检查项 | 结论 |
|---|---|
| ① `dry_run=True` 是否绝对不持久化写入 | **`db.rollback()`（L296）在返回前回滚所有 pending** → 不持久化。但需注意：在回滚前，`rebuild=True` 路径仍会向连接**发出** `EventOpinion.delete()`（L397）与 `db.add/flush` 语句（仅被回滚，未落盘）。 |
| ② `rebuild=True` 是否只影响计算范围 | 是。仅 `_reset_active_event_links()` 清理“活跃事件”关联，随后从候选池重算；**不删 Event 行、不碰陈旧事件**。 |
| ③ 现有 Event / EventOpinion 是否只读 | dry-run 下回滚，等效只读。 |
| ④ 能否完整生成新 Event 候选结果 | 能（本脚本用同款纯函数 `cluster_opinions` + 同款物化/匹配逻辑复现）。 |
| ⑤ 能否获得旧→新成员映射 | 能（shared-member 优先 + continuation 匹配）。 |

**为避免“向连接发出任何写语句”的边界风险，本阶段改用纯只读模拟**：仅 `SELECT` 加载 ORM 对象，
调用真实的 `cluster_opinions` / `_merge_condition` / `_match_existing_event` 等价逻辑，在内存中完成全部预测。
这比调用 `aggregate(dry_run=True)` 更严格地满足“禁止任何写操作”。

配置阈值（来自 `config.py`，Phase 4-Event-1 新增，未改动）：
`event_window_days=7`、`event_text_similarity_threshold=0.45`、
`event_low_merge_text_threshold=0.30`、`event_continuation_days=14`、
`event_continuation_text_threshold=0.35`、`event_singleton_min_risk=40`。

## 5. 单成员 Event 语义分析

| 指标 | 新规则值 |
|---|---|
| 单成员 Event 数 | 59 |
| 多成员 Event 数 | 25 |
| 单成员占比 | **70.2%** |

**判断（依据暂定业务定义：Event = 面向研判的事件聚合单元，可由一条高风险 Opinion 独立形成，也可由多条相关 Opinion 聚合形成）：**

- 单成员 Event **不自动视为错误**。59 个单条中，绝大多数是 `risk_score ≥ 40` 的独立高风险舆情
  （如“网传群体聚集谣言被辟谣”risk=100、“交通事故造成一人死亡”risk=100），符合“单条高风险 Opinion 独立成研判单元”的定义。
- 但 70.2% 的单成员占比偏高，提示事件中心将呈现大量“1 条舆情”事件。
  **这不影响正确性，但影响产品观感**——Phase 4-Event-2/3 的标题与摘要生成、
  时间线聚合将显著改善这类事件的“看起来像一条 Opinion”问题。
- 结论：**当前单成员 Event 符合业务定义，无需修改规则**；产品体验优化留待 Event-2/3。

## 6. 规则质量校验（链式 vs 高相似保留）

| 校验项 | 结果 |
|---|---|
| 旧多成员 Event 中仍被新规则保留部分同簇 | 8 |
| 旧多成员 Event 中被新规则打散（无 ≥2 同簇） | 9 |
| pool 内文本相似度 ≥0.80 的对 | 24 |
| 其中仍被聚为同簇 | **15（62.5%）** |
| 其中因 region/time 门槛离散 | 9（37.5%） |

**关键发现（需产品/架构决策）：**
- 15/24 高相似对保留 → 规则**未过严**，真实同事件内容仍聚合。
- 9/24 高相似对离散，**原因仅为 region_id 不同 或 时间超窗**（文本相似 ≥0.80 必然 ≥0.45 阈值，
  故只可能被 region/time 硬门槛阻断）。
- `region_id` 当前是**省/市/县行政区划**（非精确地点）。同一真实事件在跨区转载/多源分发时，
  region 不同会导致**合理同事件被拆开**（过度拆分）。这是当前最值得在 Phase 4-Event-4 评估的敏感性：
  **当文本相似度极高（如 ≥0.80）时，是否应将 region 由硬门槛降级为软加权信号。**
- 不存在“所有 Event 都变单条”（规则过严）或“大量无关仍合并”（规则过松）的极端情况。

## 7. 正式 Rebuild 风险评估（⚠️ 重点）

评估若在生产库执行 `aggregate(rebuild=True)`（非 dry-run）的影响：

| 风险项 | 评估 |
|---|---|
| ① Event ID 是否变化 | **会**。rebuild 重置活跃事件关联后重算；45 个旧事件变空、20 个新 Event 生成新 ID。 |
| ② EventOpinion 关联是否变化 | 是。活跃事件关联被 `_reset_active_event_links` 先删后重挂。 |
| ③ 旧 Event 是否会被删除 | **当前代码不删 Event 行**（仅删关联）。 |
| ④ 是否会产生新 Event | 是，预测 +20 个。 |
| ⑤ 是否造成历史关联丢失 | **是，且更严重**：45 个活跃旧事件链接被清后**无新成员挂回 → 成为空 Event 行**（opinion_count 仍是旧值、links=0）。 |
| ⑥ 是否存在回滚方案 | **当前无**。代码无备份/快照/事务外回滚机制。 |
| ⑦ 是否能保留旧结果审计记录 | **否**。旧 Event/关联被覆盖，无归档。 |

**严重结论：**
> 当前 `rebuild=True` 实现，在**更严格的新规则**下，会把 70 个旧事件中的
> **45 个变成空孤儿 Event**（链接清空、计数失真），并将总数从 70 **膨胀到约 90**，
> 且**无备份、无回滚、无审计归档**。

依据本阶段禁令“如果正式 rebuild 会直接删除旧 Event 且没有备份/回滚机制：不要执行正式 rebuild”，
**此处进一步判定：即便不删除，孤儿膨胀 + 无回滚同样不可接受 → 绝不在本阶段执行正式 rebuild。**

## 8. 是否建议正式执行 Rebuild

**建议：否（本阶段不执行）。**

理由：
1. 当前 `rebuild=True` 实现存在 §7 的孤儿膨胀缺陷，且无备份/回滚。
2. 更优的迁移路径是 **“备份 → 全量清空 event/event_opinion → 全新 aggregate（非 rebuild）”**，
   但这涉及 DELETE，必须单独作为受控迁移任务（建议归入 Phase 4-Event-4），并先建备份表。
3. 新规则本身已通过 dry-run 验证为正确（§4、§6），无需借 rebuild 验证。
