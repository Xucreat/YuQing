## 0. TL;DR（结论先行）

1. **项目已有很好的基础，远非"白纸"。** `BaseCollector` 抽象、`CollectorService` 的「采集→去重→建 Opinion→AI 分析」闭环、`APScheduler` 调度、`Region` 模型（已带 `province/city/county/...` 层级字段）、统一去重与 AI 写回——这些都**不应推翻**，应直接复用。

2. **参考方案里的 `collectors/xinhua.py`、`people.py` 等分文件写法，在企业级"省→市→县"规模下会演变成大量重复代码。** 更优方案是：**抽象一个 `BaseHttpCollector`（共享 HTTP/解析/限流/关键词过滤）+ 一个参数化的 `GenericSiteCollector`**，新站点（新华、人民、中新、河北日报、长城网、各市政府网）几乎只写"配置"，不写"代码"。

3. **真正缺失的不是"采集器类"，而是"装配与编排层"。** 项目里 `BaiduNewsCollector`、`HebeiNewsCollector` 已经写好且是**真实 HTTP 抓取**，但 `resolve_collectors()` 里一段 `if/else` 只装配了 `GovernmentCollector`，其余 3 个**根本没被调用**。同时 `config.py` 里的 `baidu_news_enabled` / `hebei_news_enabled` 等开关是**死配置**（全仓库无人读取）。

4. **区域被硬编码为"大厂县 131028"。** `CollectorService._resolve_region_id` 写死查 `Region.code == "131028"`。要支持"河北全省 + 全国"，第一步是**区域数据化 + 按采集器绑定区域**，而不是改一行硬编码。

5. **`CollectorRun` 模型已定义但从未写入**，`/sources/history` 永远返回空。采集历史/失败审计目前只存在于内存字典，重启即丢。

**一句话推荐**：不要新建一堆采集器文件，而是**把已有能力"接上 + 抽薄 + 数据驱动"**——注册表 + 共享解析基类 + 区域数据化 + 让死配置生效。这是改动最小、最可持续的方向。

## 1. 当前架构总结

### 1.1 数据流（现状）
```
APScheduler (cron */30)  ─┐
POST /api/collector/run  ─┼─► CollectorService.collect_and_analyze(db)
                           │      │
                           │      ├─ resolve_collectors() ─► [GovernmentCollector]   ← 只装了这一个
                           │      ├─ for collector: collector.fetch()  ─► list[dict]
                           │      ├─ 按 url 去重 (_already_exists)
                           │      ├─ 建 Opinion(region_id=131028 硬编码, analysis_status=pending)
                           │      └─ AIService.analyze() 写回 summary/sentiment/risk_score/keywords

```

### 1.3 统一契约（已存在，应保留）
- `fetch() -> list[dict]`，dict 字段约定：`{title, content, source, url, publish_time}`。
- 采集器**禁止直接写库**：`Collector.fetch() -> Service -> Database`。这是好约束，延续。
- 去重以 `url` 为准（空 url 退回 `title+publish_time`）。
- AI 分析统一走 `AIService`，单条失败隔离（`failed` 状态保留记录）。

## 5. 发现的问题（按严重度）

- **P0 · 装配断裂**：`resolve_collectors` 只返回 `GovernmentCollector`，已写好的 `BaiduNewsCollector`/`HebeiNewsCollector` 永不运行。等于"有数据源但没接上"。
- **P0 · 区域硬编码**：`_resolve_region_id` 写死 `131028`，无法表达"河北全省"或多区域同时监测。
- **P1 · 死配置**：`baidu_news_enabled`/`heibo_news_enabled`/`weibo_enabled` 无任何读取方，配置形同虚设，"启用/停用"能力失效。
- **P1 · 解析重复**：`GovernmentCollector._parse_detail` 与 `HebeiNewsCollector._parse_content` + `_CONTENT_SELECTORS` 逐字相同，规模化到"11 市 + 各县"会严重复制。
- **P2 · 采集历史缺失**：`CollectorRun` 不写入，`/sources/history` 恒空；运维无法审计失败/频率。
- **P2 · 限流/重试硬编码**：`REQUEST_INTERVAL`/`MAX_ARTICLES` 写死在各类里；无重试、无退避；规模化后有封禁风险（且项目明确禁止代理/反爬，更需靠"礼貌限速 + 重试"兜底）。
- **P2 · 关键词全局**：`.env` 一行 CSV 全局生效，无法按区域/按源差异化（但**当前阶段不建议引入 per-region 关键词**，避免过度设计）。

## 6. 推荐架构（优于参考方案）

### 6.1 为什么参考方案的 `collectors/xinhua.py` 等分文件写法不是最优

参考方案为新华、人民、中新各写一个采集器文件。这些站点**结构高度相似**（列表页 → 详情页 → 正文容器选择器）。若每个省/市/县政府网都照此写一个文件，河北 11 市 + 各县就是几十个近乎重复的文件，维护成本爆炸，且违背本项目已有的"共享解析"意图（`HebeiNewsCollector` 已照搬 `GovernmentCollector` 的选择器即佐证）。

### 6.2 推荐：两层抽象 + 注册表 + 数据驱动

```
BaseCollector (已有, 不动)
   │  fetch() -> list[dict]
   ▼
BaseHttpCollector (新增, 抽共享能力)
   - 共享 requests Session + 桌面 UA
   - _get(url, retries=N, backoff=...)   # 带可配置重试/超时
   - extract_content(html) -> str         # 复用现有 _CONTENT_SELECTORS 降级逻辑
   - match_keywords(text) -> bool         # 关键词过滤
   - rate_limit()                         # 读取配置间隔
   ▼
GenericSiteCollector (新增, 参数化"列表→详情"站点)
   - 构造参数: list_urls, article_link_selector / article_link_regex,
               content_selectors, scope_region_codes, source_name, priority
   - 自动完成: 列表取链 → 去重 → 限速抓详情 → extract_content → 关键词过滤 → 返回 dict
   ▼
具体源 (薄子类 / 纯配置):
   GovernmentCollector      (已有, 迁为 GenericSiteCollector 配置)
   HebeiNewsCollector       (已有, 迁为 GenericSiteCollector 配置)
   XinhuaCollector          (新华网, 配置: news.cn 列表/详情规则)
   PeopleCollector          (人民网, 配置)
   ChinaNewsCollector       (中新网, 配置)
   HebeiDailyCollector      (河北日报, 配置)
   GreatWallCollector       (长城网, 配置)
   HebeiGovCollector        (河北省政府, 配置)
   BaiduNewsCollector       (已有, 搜索型, 可保留为特例不强制并入 Generic)
   ── 未来市级/县级 ──
   GenericSiteCollector(urls=某市/县政府网, region=该市/县)  # 一行配置加一个源
```

**注册表（替代 if/else）**——放在 `collectors/registry.py`：
```python
COLLECTOR_REGISTRY: dict[str, "CollectorMeta"] = {}
def register(key, cls, *, enabled=True, priority=100,
             scope_region_codes=None, extra=None):
    COLLECTOR_REGISTRY[key] = CollectorMeta(...)

# 在各采集器模块底部声明（或集中声明）：
register("government", GovernmentCollector, enabled=True, priority=10, scope_region_codes=["131028"])
register("baidu_news", BaiduNewsCollector, enabled=settings.baidu_news_enabled, priority=20)
register("hebei_news", HebeiNewsCollector, enabled=settings.hebei_news_enabled, priority=30, scope_region_codes=["130000","131000"])
register("xinhua", XinhuaCollector, enabled=False, priority=5)   # 先关，验证后开
...
```
`resolve_collectors(region_code=None)` 改为：**读注册表 → 过滤 enabled → 按 priority 排序 → 按 region_code 过滤 scope → 实例化**。这样"新增数据源 = 写一个薄子类/配置 + 一行 `register`"，**中心代码零改动**。

### 6.3 优点 / 缺点 / 扩展性

| 维度 | 评价 |
|---|---|
| 优点 | ① 复用已有闭环与调度，零重写；② 新站点几乎只写配置，杜绝复制；③ 注册表让"启用/停用/优先级/区域"数据驱动；④ 向后兼容（`GovernmentCollector` 行为不变） |
| 缺点 | ① `GenericSiteCollector` 对"非标准列表→详情"站点（如百度搜索）需特例处理；② 注册表是个新约定，需团队遵守；③ 初期需把现有 2 个采集器"迁"到新基类（行为须保持不变，需测试守护） |
| 扩展性 | **省→市→县→源**只需加 `register` + 区域种子；**扩展到全国**只需加省级 `Region` + 国家级源（`scope_region_codes=None` 表示全国），**架构不变** |

### 6.4 区域绑定设计（关键，支撑省→市→县）

- 每个采集器声明 `scope_region_codes`（它"关于"哪些区域）。例如 `HebeiNewsCollector` 覆盖 `["130000","131000"]`（省+廊坊），`GovernmentCollector` 覆盖 `["131028"]`（大厂县），国家级源覆盖 `None`（全国）。
- `CollectorService` 把每条采集到的 Opinion 的 `region_id` 设为**该采集器 scope 中"最具体匹配当前监测目标"的区域**（或直接取采集器主区域）。
- **不改动 `Opinion` 表结构**，`region_id` 字段已存在。
- 监测目标从"硬编码 131028"变为"配置一组要监测的 region_codes"，可含省、市、县多层。

> 说明：当前 `GovernmentCollector` 仍绑定大厂县，**完全兼容**；只是"绑定方式"从硬编码改为声明式，且支持多区域同时运行。

## 7. 河北省级数据源体系设计（省→市→县→数据源）

### 7.1 区域数据（种子化，写入 `regions` 表）
```
河北省 (130000, province)
├─ 石家庄市 (130100, city)
├─ 唐山市 (130200, city)
├─ 秦皇岛市 (130300, city)
├─ 邯郸市 (130400, city)
├─ 邢台市 (130500, city)
├─ 保定市 (130600, city)
├─ 张家口市 (130700, city)
├─ 承德市 (130800, city)
├─ 沧州市 (130900, city)
├─ 廊坊市 (131000, city)
│   └─ 大厂回族自治县 (131028, county)   ← 已有
├─ 衡水市 (131100, city)
└─ 雄安新区 (139000, city) 等
```
（仅初始化"要监测"的部分；`level` 字段已支持，无需改表。）

### 7.2 源与区域映射（注册表 + scope_region_codes）
| 层级 | 数据源 | scope | 实现方式 |
|---|---|---|---|
| 国家级 | 新华网、人民网、中国新闻网、百度新闻 | 全国(None) 或 含河北 | `GenericSiteCollector`/已有 Baidu，配置 |
| 省级 | 河北新闻网、河北日报、长城网、河北省政府 | `130000` | `GenericSiteCollector` 配置 |
| 市级 | 11 市新闻网/政府网 | 各市 code | `GenericSiteCollector` 配置（同一基类，零新代码） |
| 县级 | 各县政府网（含大厂县） | 各县 code | `GovernmentCollector`/`GenericSiteCollector` 配置 |

加 `province` 级 `Region`（如北京 110000、山东 370000…）+ 对应省级源即可。`CollectorService` 已按"每采集器绑定区域"工作，无需改架构。
