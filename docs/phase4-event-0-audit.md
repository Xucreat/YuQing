# Phase 4-Event-0 Event Aggregation Rule Audit

> 阶段定位：**分析与方案设计阶段（只读，未改动任何代码/表结构/API/前端）**。
> 数据来源：`postgresql://opinion_user:opinion_pass@127.0.0.1:5432/opinion_db` 只读查询（脚本见文末附录）。
> 结论先行：**当前 `EventAggregator` 是可用但"伪聚合"倾向严重的 MVP 规则实现；根因不在聚合算法本身，而在 `Opinion.keywords` 这个"唯一聚合信号"覆盖率极低且区分度极差。** 推荐采用"规则召回 + 相似度剪枝 + AI 叙事"的分层方案。

---

## 1. 当前聚合规则总结

入口：`POST /api/events/aggregate` → `EventAggregator().aggregate(db)`（`backend/app/services/event/aggregator.py`）。

1. **取数**：`Opinion` 中 `analysis_status == "completed"` 且 `keywords` 非空 且 `created_at >= now - event_window_days`（默认 **7 天**）。
2. **分桶**：按 `region_id` 分组，跨区域永不合并。
3. **组内聚类**：同 region 内，两两比较 `keywords`（逗号拆词），**任一交集非空即 union（并查集连通分量）**。复杂度 O(n²)。
4. **关键词并集**：每个 cluster 的 `keywords` 取并集。
5. **既有 Event 匹配**：用并集与所有 `Event.keyword` 做交集，命中**首个**即视为同一事件。
6. **创建/更新**：无命中→新建 `Event`；有命中→显式 `EventOpinion` 关联（已关联跳过）并重算派生字段。
7. **派生字段**：
   - `title` = 聚类中 `risk_score` 最高的 `Opinion.title`
   - `description` = 该最高风险舆情 `content` 前 200 字
   - `keyword` = 聚类内 `keywords` 并集（排序逗号拼接）
   - `risk_level` = 聚类内最大 `risk_score` 映射（≥70 high / ≥40 medium / else low）
   - `opinion_count` = 关联舆情数
   - `first_time` / `last_time` = 关联舆情 `created_at` 最小/最大
8. **副作用**：对每个新建/更新的 Event 自动触发 `PropagationService.rebuild_for_event`。
9. **返回**：`{created, updated, linked}`。

约束：仅通过 `EventOpinion` 显式建关联；`Event` 无 `status` 列（API 固定返回 `"active"`）；不改表结构。

## 2. Opinion.keywords 真实语义

**确认结论：`Opinion.keywords` 的真实语义是"文本命中的风险/舆情主题标签（来自内置敏感词表）"，既不是监测词表、也不是 AI 抽取关键词。**

生成链路（代码实证）：
- 采集阶段 `CollectorService`（`backend/app/collectors/service.py:203`）**直接**使用 `RuleFallbackProvider()`（不走 `AIService`，**不调用 DeepSeek**）。
- `analysis = RuleFallbackProvider().analyze("标题：" + title + "\n正文：" + content)`。
- `RuleFallbackProvider`（`backend/app/services/ai/fallback.py`）持有**内置 `DEFAULT_KEYWORDS`（16 个固定词+权重）**：火灾、爆炸、事故、伤亡、死亡、冲突、群体、上访、维权、投诉、谣言、诈骗、腐败、贪污、涉警、舆情。
- `analyze` 把"标题+正文"对这 16 个词做**字面 `word in text` 包含判断**，命中的词记入 `hits`。
- 回写：`opinion.keywords = ",".join(analysis.keywords)`（service.py:258）。

必须明确的三个"不是"：
- ❌ **不是** `keywords` 监测词表（该表仅用于采集过滤 `matches_keywords` 与预警匹配，由 `keyword_service.get_monitoring_keywords` 读取，从未进入 `Opinion.keywords` 生成）。
- ❌ **不是** DeepSeek AI 抽取关键词（AI 结果写入的是独立字段 `ai_keywords`，且**仅当用户手动"触发 AI 分析"时**才生成）。
- ❌ **不是**该事件的唯一关键词 / 主题词——它只是"这篇文章里出现了哪些内置敏感词"。

> 注：`CollectorService` 构造 `RuleFallbackProvider()` 时未注入参数，因此**永远使用内置 16 词**，不会从数据库 `keywords` 表加载。`AIService` 的 `keywords` 注入参数当前也始终为 `None`（"未来可由外部从 keywords 表加载"尚未实现）。

## 3. 当前规则的主要问题

### 3.1 信号源覆盖率极低（最致命）
`Opinion.keywords` 仅来自 16 个内置词。任何不含这些词的舆情（会议、经济、政策、人事、民生常规等）`keywords` 为空，**直接被聚合器第 3 步（keywords 非空）整条排除**。真实数据中 **84.4% 的 Opinion（362/429）keywords 为空**，即事件中心只对 15.6% 的数据可见。

### 3.2 关键词过于粗粒度 → 伪聚合（False Merge）
"火灾/事故/投诉/维权/涉警"等词可出现在大量互不相关的事件中。当前规则：**共享任意一个通用词即合并**。这与"属于同一个现实事件"不是等价关系。

### 3.3 连通分量链式扩散
A(火灾,事故) ↔ B(事故,投诉) ↔ C(投诉,维权) → A+B+C 被并查集合并。A 与 C 可能毫无直接语义关联，却因"事故/投诉"作为桥被拉进同一 Event。这是一个**传递闭包式误合并**，且无法解释"为什么 A 和 C 在一起"。

### 3.4 region_id 约束不充分
`region_id` 是**必要非充分**条件。同一 region 内不同时间、不同地点、不同主体、不同事件的舆情，只要都命中某个通用词（如"事故"）就会被合并。region 不能单独承担"同一事件"判定。

### 3.5 7 天窗口造成事件生命周期断裂
仅取 `created_at >= now-7d`。一个持续 10 天的事件：第 8 天起，第 1 天的舆情滑出窗口不再参与聚合；重跑时旧成员被排除，新成员可能基于"剩余关键词"形成**新的独立 Event**，导致同一现实事件被拆成多个、且旧 Event 与新 Event 间无延续关联。

### 3.6 Event.title 语义错位
`title = 最高 risk_score 的 Opinion.title`。这实际上是**"代表性舆情标题"**，不是"事件标题"。当 Event 聚合了 23 条舆情时，title 却是一条具体新闻的标题——Event 看起来像"一条被放大的 Opinion"，而非一个聚合事件。

### 3.7 Event.description 语义错位
`description = 最高风险 Opinion.content 前 200 字`。这是**"代表性舆情正文片段"**，不是"事件简介/摘要"。用户无法据此理解"这个事件整体在讲什么"。

### 3.8 既有 Event 匹配取"首个命中"欠稳
`_find_existing_event` 遍历所有 Event 取第一个 `keyword` 交集命中的，无时间/规模权重，可能把新舆情误挂到陈旧或错误的 Event 上，进一步放大链式污染。

## 5. 典型反例分析

**案例 A：共享通用风险词（两起火灾）**
- A：标题"某地发生火灾"，keywords=火灾；B：标题"另一地发生火灾"，keywords=火灾。
- 当前规则（若 region_id 相同）：合并为同一 Event。
- **判断：不应简单合并。** 不同地点、不同起火原因、不同责任主体的两起火灾是**两个独立现实事件**。仅共享"火灾"这一通用词不足以判定同一事件。→ 印证 §3.2 伪聚合。

**案例 B：链式聚合**
- A(火灾,事故) ↔ B(事故,投诉) ↔ C(投诉,维权)。
- 当前规则：A、B、C 全部并入同一 Event，A 与 C 无直接共同词。
- **判断：不合理。** 这是典型的传递闭包污染。应为 3 个（或至少 2 个）独立事件。→ 印证 §3.3。真实数据 id=40/95 即此类。

**案例 C：同区域不同事件**
- A：region_id=1，标题"某工厂发生爆炸"，keywords=爆炸,事故；B：region_id=1，标题"某道路发生交通事故"，keywords=事故。
- 当前规则：region 相同且共享"事故" → 合并。
- **判断：region_id 不足以区分。** 工厂爆炸与道路事故是不同事件。region 只能缩小范围，必须叠加"具体词 + 时间 + 文本相似度"等判别器。→ 印证 §3.4。

**案例 D：同一事件跨时间持续（10 天）**
- 第 1 天"某工厂发生爆炸"、第 5 天"事故调查进展"、第 10 天"责任认定公布"。
- 当前规则：仅取 `created_at >= now-7d`。若在第 10 天重跑：第 1 天的舆情已滑出窗口，不再参与；第 5、10 天的舆情可能基于剩余关键词形成新的独立 Event，与原 Event 无延续。
- **判断：会产生事件生命周期断裂 / 同一事件被拆多份。** → 印证 §3.5。当前库无跨 7 天事件，故未观测到，但规则上必然发生。

## 6. 候选方案对比

| 方案 | 准确性 | 可解释性 | 实现复杂度 | 性能 | 推荐度 |
|---|---|---|---|---|---|
| **A. 改进规则聚合** | 中（缓解但未根治） | 高（全规则可追溯） | 低 | 高（O(n²) 可控） | ⭐⭐⭐ 作为基础必做 |
| **B. 文本相似度聚合（TF-IDF/词频余弦）** | 高 | 中（可展示相似度分） | 中 | 中（可纯 PG `pg_trgm`/Python 计算） | ⭐⭐⭐⭐ 强烈建议叠加 |
| **C. Embedding / AI 语义聚类** | 很高 | 低（向量不可读） | 高 | 依赖外部 API/算力 | ⭐⭐ 暂不主推 |
| **D.（推荐）分层：规则召回 + 相似度剪枝 + AI 叙事** | 高 | 高（成员规则定、叙事 AI 写） | 中 | 中 | ⭐⭐⭐⭐⭐ 推荐落地 |

**方案 A（改进规则聚合）**
- 要点：保留 region + 时间窗；`keywords` 仅作候选召回；**禁止仅凭单一通用词合并**；为通用词（事故/投诉/舆情/群体等）降权，要求 ≥2 个关键词共同命中或含 ≥1 个"特异性词"；限制连通分量链式扩散（如最大跳数/要求存在"锚点词"被多数成员共享）；增加时间距离约束（相邻成员时间差上限）；`title`/`description` 改为基于主题词生成而非直接取最高风险舆情。
- 优点：零新基础设施、可解释、改动小、契合 MVP→扩展演进。
- 缺点：关键词体系本身稀疏（16 词/84% 空），单靠规则天花板明显，仍会漏聚大量空 keywords 舆情。
- 适合度：当前阶段**必须做**，但不够。

**方案 B（文本相似度聚合）**
- 要点：用 `title+content` 计算 TF-IDF / 词频余弦相似度（可纯 Python，或 PG 内置 `pg_trgm`/`similarity`），仅当相似度超阈值才合并；`keywords` 仅用于粗召回以降计算量。
- 优点：直接刻画"是否同一事件"，显著抑制伪聚合与链式扩散；不引入新服务（Postgres-only 友好）；阈值可解释。
- 缺点：需中文分词/特征工程；阈值需调参；运算量高于纯规则。
- 适合度：与本项目高度契合，是规则之后性价比最高的提升。

**方案 C（Embedding / AI 语义聚类）**
- 要点：用 embedding 算语义相似度聚类，或让 AI 直接判定归并。
- 优点：语义层面最准。
- 缺点：调用成本与延迟、可解释性差（向量不可读）、模型/外部依赖、离线能力与数据隐私风险、生产稳定性需额外保障——与"当前不希望过早引入复杂基础设施"相悖。
- 适合度：作为**可选增强**（如用已配置的 DeepSeek 做事件级标题/摘要生成），不宜作为成员判定主路径。

**方案 D（推荐·分层）** = A 的确定性召回 + B 的相似度剪枝 + C（仅限已配置的 DeepSeek）的**事件级叙事生成**：
- 成员判定（谁属于事件）→ **规则 + 相似度**，确定性、可解释、可回放；
- 事件表达（title/description/summary）→ **AI 生成**（DeepSeek 已集成，且 `ai_*` 字段机制已就绪），把"为什么聚在一起"用自然语言讲清楚。
- 完全 Postgres-only，无 Redis/ES/MQ；可分阶段演进；既提升准确性又补上当前最缺的"可解释性"。

## 7. 推荐方案

**结论：采用方案 D（分层），并立即实施其中的方案 A 作为底座。**

理由（结合项目实际）：
1. **企业级 + 持续扩展 + 约 22 数据源 + 未来更大体量** → 需要可演进、可解释的架构，而非一次性的"聪明算法"。
2. **PostgreSQL only，无 Redis/ES/MQ/Celery** → 方案 C 的向量库/队列不现实；方案 B 的相似度可在 PG/Python 内完成，零新增设施。
3. **当前最大痛点是信号源quality**（84% 空、16 词、通用词主导），**不是"要不要上 AI"**。因此：
   - **第一步必做（方案 A）**：修复 `keywords` 信号——把"可作为聚合信号"的来源从"仅 16 内置词"扩展为 **内置词 ∪ 监测词表命中 ∪（可选）ai_keywords**；并对通用词降权、禁单一通用词直并、限链式扩散、加时间距离。这是低成本高收益的地基。
   - **第二步叠加（方案 B）**：在规则召回的候选 cluster 上做 `title+content` 相似度计算，低于阈值则**拆分**（剪枝伪聚合/链式），高于阈值才定案。把"可解释"从"共享了哪个词"升级为"文本相似度 0.xx"。
   - **第三步用 AI 补叙事（方案 C 的受控使用）**：事件定案后，用**已配置的 DeepSeek** 生成 `title`/`description`/`summary`（写入新增的叙事字段或复用现有字段），解决 §3.6/§3.7 的语义错位。DeepSeek 已就绪、`ai_*` 字段范式已验证，成本可控、离线可降级。
4. **不推荐**纯规则止步（天花板低）或一步到位上 embedding 集群（过度工程、不可解释、依赖重）。
5. **时间窗口**改为"滑动 + 事件延续"：新建事件仍只从近 N 天舆情发起；但**新舆情可基于与既有 Event 关键词/相似度重叠被续挂**，避免 §3.5 的生命周期断裂。

一句话：**用规则+相似度决定"谁和谁是一件事"（确定、可回放），用 AI 决定"这件事怎么用一句话讲清楚"（可读、可解释）。**

## 8. Event 业务语义重新定义

**推荐定义：Event = 关于同一现实事件的一组 Opinion（候选定义 B），是一个面向研判的聚合单元，其成员由确定性规则+相似度判定，其表达由 AI 生成的可读叙事承载。**

对比候选：
- A（共享风险关键词的一组 Opinion）：当前实现，**错误**——把"共享标签"当"同一事件"，导致伪聚合。
- B（关于同一现实事件的一组 Opinion）：**推荐**——以现实事件身份为锚。
- C（关于同一主题的一组 Opinion）：可作为 Event 的"子视图/标签"，不宜作为 Event 唯一身份。
- D（需要研判的舆情聚合单元）：是 Event 的**用途**描述，可并存，但不替代 B 作为身份定义。

字段语义应重定义为：

| 字段 | 当前（问题） | 应表示 |
|---|---|---|
| `Event.title` | 最高风险 Opinion 的原始标题（像一条新闻） | **事件主题/标题**：由"高频非通用主题词 + 代表性舆情"经规则或 AI 生成，如"某工厂爆炸事故及后续调查" |
| `Event.description` | 某条舆情正文前 200 字 | **事件简介/摘要**：聚合多条舆情核心议题，可由 AI 基于成员生成 |
| `Event.keyword` | 聚类 keywords 并集（与监测词混淆） | **该事件的代表性主题标签**（区分于 `keywords` 监测词表）：取聚类内高频且**非通用**的词 |

## 11. 风险与待确认事项

1. **信号源扩展是否引入监测词表到 `Opinion.keywords`？** 若把监测词表命中并入 `keywords`，会改变"keywords=内置敏感词"的现有语义，需确认是否与预警/采集过滤逻辑冲突（当前二者共用 `keywords` 表，但写入路径独立）。
2. **相似度方案的中文分词依赖**：是否需要引入 `jieba` 等分词（增加依赖），还是直接用字符级 `pg_trgm`/`TF-IDF`（零依赖、够用）？建议先零依赖验证。
3. **AI 叙事的生成成本与频率**：每次聚合/每次事件变更都调 DeepSeek 可能累积成本；建议"事件级去重生成 + 缓存 + 失败降级规则"。需确认 DeepSeek 额度与调用策略。
4. **既有 70 个 Event 的处理**：新规则上线后旧 Event 是否保留/重算/清空重建？建议提供"重新聚合（清空重建）"开关并保留历史可审计。
5. **`ai_keywords` 当前全库为空（0 条）**：AI 信号从未生成，Phase 4-Event-1 若依赖 `ai_keywords` 需先有数据；建议首版仅以"内置词 ∪ 监测词表"为主信号。
6. **时间窗口与事件延续的边界**：滑窗长度、续挂的相似度/重叠阈值需结合真实数据标定（Phase 4-Event-4 完成）。
7. **可解释性呈现程度**：是否在前端直接展示"相似度分数 + 共现词"作为聚合依据？影响前端工作量，需产品确认。
