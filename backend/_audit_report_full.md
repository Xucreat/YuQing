→ 合并是**数据无损**操作（合并后信息量 = 合并前并集，且无新增噪声）。

## 3. 聚合代码审计（并发窗口）

审计对象：`backend/app/services/event/aggregator.py`

### 3.1 调用链

- `auto_aggregate_after_collect(session_factory)`（L682）：采集完成后自动增量聚合，**每次新建独立会话**（`db = session_factory()`，L695），异常安全不抛。
- `EventAggregator.aggregate(..., incremental=True)`（L246→L268）：转发到 `_aggregate_incremental`。
- `_aggregate_incremental(db, dry_run)`（L360）：核心增量逻辑。

### 3.2 并发窗口（根因）

1. **候选召回基于已提交状态**（L376-385）：`Opinion.id NOT IN (event_opinions)` 仅读已提交的关联，无 SERIALIZABLE、无行锁。
2. **物化前不复核既有事件**（L462-479）：`unattached` 候选经 `cluster_opinions` 后直接 `_create_event(db, cluster)`（L475），**不走 `_match_existing_event`**（该函数仅用于全量路径，L514）。
3. **提交在最后**（L492）：两次并发调用都在各自事务内读到同一批未关联舆情 → 各自物化为新事件 → 同一簇被建两次，标题因代表舆情/成员数相同而一致。

### 3.3 是否存在事务保护 / 锁

- 全文检索 `advisory / with_for_update / lockmode / SERIALIZABLE / Lock / threading / asyncio.Lock`：**0 命中**。
- `_create_event`（L597+）：仅构造 `Event` 并插入，无任何去重前置检查。
- 增量路径的“防重”只靠 `attached` 环节用 `_merge_condition` 匹配**活跃**事件（L437-452）；但并发第二调用在读取时刻第一调用尚未提交，活跃事件与关联均不存在 → 防御失效。

### 3.4 是否适合增加 advisory lock

**适合，且为本代码库已验证范式**：`app/core/scheduler.py` 已用 `pg_try_advisory_lock` 实现跨进程单例（Phase 7 落地并验收通过）。

- 实现要点：在 `aggregate()`（或 `_aggregate_incremental`）入口对**本次聚合使用的同一会话/连接**取 `pg_try_advisory_lock(固定key, e.g. hash('event-aggregate-singleton'))`；取不到则跳过（返回 created=0）；`finally` 中释放。
- 必须取在与候选召回/提交**同一连接**上，才能真正串行化两个并发聚合事务（锁在别的连接不生效）。`auto_aggregate_after_collect` 的 `session_factory()` 单会话恰好满足。
- 零新依赖、不引入 Redis、与现有 scheduler 锁同源、可跨进程（杜绝多实例同时聚合）。

## 4. 解决方案

### 4.1 短期：阻止继续产生重复事件（优先级最高）

- **S1 聚合串行化锁**：仿 `scheduler.py` 在 `EventAggregator.aggregate` 入口加 advisory lock（见 3.4），1 处改动。立即消除并发双物化。
- **S2 收口冗余实例**：关停 `:8011` 后端实例（Phase 7 已识别但未关停），与 S1 叠加彻底消除多进程并发源。
- **S3 手动聚合并发提示**：前端「手动聚合」按钮在聚合进行中置灰/禁用，规避人工双触发（低成本补充）。

### 4.2 中期：安全合并历史重复事件（事务化，遵循 Phase 7 纪律）

1. **备份**：`pg_dump -Fc opinion_db` 落盘（不跳过备份）。
2. **事务内执行合并脚本**（伪代码，单事务、验证后 COMMIT）：
   ```sql
   BEGIN;
   -- 对每组：保留 keep，删除冗余 R 的子表行（opinions 已挂 keep，无丢失）
   DELETE FROM event_opinions      WHERE event_id = ANY(:redundant_ids);
   DELETE FROM propagation_nodes    WHERE event_id = ANY(:redundant_ids);
   UPDATE alert_records SET event_id = keep WHERE event_id = ANY(:redundant_ids); -- 本批 0 行
   DELETE FROM events              WHERE id = ANY(:redundant_ids);
   -- 验证：重复组数应为 0
   SELECT count(*) FROM (SELECT title,count(*) FROM events GROUP BY title HAVING count(*)>1) t; -- 期望 0
   COMMIT;  -- 验证非 0 则 ROLLBACK
   ```
3. **重建传播树**：对 43 个保留事件逐一 `rebuild_for_event(keep_id)`，消除镜像重复节点。
4. **验证**：重复事件组=0、`propagation_nodes` 悬空父引用=0、`alert_records` 仍=3、总事件数=194。

> 纪律红线（沿用 Phase 7）：合并前必须 SELECT 验证保留侧 opinion 集合覆盖冗余侧（A 类恒成立）；不直接 DELETE 未验证数据；不自动删业务数据。

### 4.3 长期：事件级幂等设计

- **L1 事件去重键 + 部分唯一索引**：对活跃事件计算规范化去重键（如 `md5(sorted(opinion_ids))` 或代表舆情 url），加 `UNIQUE` 部分索引（窗口内活跃事件），作纵深防御。
- **L2 `_create_event` 前置 `_match_existing_event`**：增量路径物化前也调用既有事件匹配（共享成员/延续），与全量路径一致，缩小逻辑竞态。
- **L3 串行化隔离级别**：评估对聚合事务采用 `SERIALIZABLE`（并发冲突时捕获 serialization failure 并重试），作为 advisory lock 的互补。
- **L4 单实例/单写部署**：将聚合、调度统一到单一写节点，从架构上消除多进程并发源。

## 5. 结论与后续技术债

- **诊断结论**：事件重复是 Phase 7 之前「多实例双触发 + 聚合无并发串行化」的系统产物，43 组全部为可无损合并的 A 类完全重复；当前唯一真因（双触发）已由 scheduler advisory lock 阻断，但存量与并发窗口待收口。
