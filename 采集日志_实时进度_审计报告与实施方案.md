## 一、任务触发链审计（task_id / batch_id / 返回值 / 竞态）

**手动采集完整链路（已逐行核实）：**

```
用户点「开始采集」
  └─ AppLayout.vue handleCollect()
       └─ POST /api/collector/run
            └─ run_collector() [collector.py:82]
                 └─ task_id = start_task("collector", _run_collect_task, SessionLocal)  ← ① 内存生成 task_id
                 └─ 返回 CollectorTaskResponse(task_id=task_id)
       └─ pollTask(task_id) 轮询 GET /api/tasks/{task_id} 至终态
       └─ 完成 → ElMessage + dispatchEvent('data-refresh')
            └─ _run_collect_task(task, session_factory) [collector.py:51]
                 └─ service.collect_and_analyze_concurrent(session_factory, on_progress=_on_progress)
                      └─ run_start = now(utc); batch_id = uuid.uuid4().hex  ← ② 内存生成 batch_id
                      └─ ThreadPoolExecutor 每源 _process_collector(cdb, …, batch_id, 'manual')
                           └─ CollectorRun(status='running') db.add+commit   ← ③ 每源实时落库
                           └─ 完成 → 更新 status/fetched/created/analyzed/end_time db.commit
                 └─ auto_aggregate_after_collect(...)  ← ④ 采集后仍跑聚合
                 └─ return collect_result（不含 batch_id）
```

**关键审计发现：**

1. **task_id 与 batch_id 当前无任何关联。**
   - `task_id` 由 `task_manager.start_task` 在内存 `_tasks` 字典生成，重启丢失。
   - `batch_id` 由 `collect_and_analyze_concurrent` 在 `service.py:391` 生成，**仅传入 `_process_collector` 写 CollectorRun，函数返回时被丢弃**（`return merged`，`merged` 是 `CollectorRunResult`，无 batch_id 字段）。
   - `Task` 数据类（`task_manager.py:39`）**无 `batch_id` 字段**；`to_dict()` 输出也不含 batch_id。
   - 结论：前端拿到 `task_id` 后，无法从 `/tasks/{task_id}` 反查「这次采集对应哪个 batch_id」。

2. **返回值链断层。**
   - `collect_and_analyze_concurrent` 返回 `CollectorRunResult`（collector_type/fetched_raw/created/analyzed/failed/aggregated）。
   - `_run_collect_task` 将其包装成 `collect_result` dict（collector.py:61-67），**仍不含 batch_id**。
   - `Task.result = collect_result`，因此 `/tasks/{task_id}` 的 `result` 也无 batch_id。

3. **竞态根因（必须正视）：**
   - 定时 cron（`scheduler._run_collector_job`）**不经过 task_manager**，无 task_id，但同样会生成 batch_id 并写 CollectorRun（trigger_type='scheduled'）。
   - 若手动采集与定时采集在时间上重叠，DB 中会出现**两个 running 批次**。此时「取最新 running 批次」的纯前端方案无法区分哪个是用户刚点的 → 进度面板可能展示错误的批次。
   - **根治办法 = 建立 task_id↔batch_id 关联**（见 §八.5）。

## 二、on_progress 能力审计

**当前实现（`service.py:446-447` + `collector.py:53-55`）：**

```python
# service.py 在 as_completed 循环里调用：
if on_progress is not None:
    on_progress(done, total, getattr(collector, "source_name", ""))

# collector.py 的回调：
def _on_progress(done, total, name):
    task.progress = int(done / total * 100) if total else 100
    task.step = f"已采集 {done}/{total} 个数据源" + (f"（{name}）" if name else "")
```

**审计发现：**

1. **on_progress 签名 `(done, total, name)` 信息极薄：**
   - `done` = 已**完成**源数（as_completed 计数），非「正在处理」数。
   - `total` = 数据源总数。
   - `name` = **刚刚完成**的那个源名（循环里 `collector` 是刚 future 完成的源），**不是当前正在处理的源**。
   - **无任何逐源实时状态、抓取条数、新增条数、分析条数、错误信息。**

2. **on_progress 只写内存 `task` 对象，不写 DB、不通知任何人。** 进度只存在于后台任务线程的内存里，只有 `GET /tasks/{task_id}` 能读到。

3. **聚合阶段语义缺失：** `auto_aggregate_after_collect` 运行时，所有 CollectorRun 已是终态，但 `task.status` 仍为 running、`task.step="采集完成，正在自动聚合事件…"`。若前端仅凭 CollectorRun 全部终态就判定「采集完成」，会与「任务仍在 running」冲突。

**结论：** on_progress 当前**不足以**支撑「当前正在处理的数据源」展示。准确来源应是 **DB 中 status='running' 的 CollectorRun**（见 §三），而非 on_progress 的 name。

## 三、CollectorRun 实时性审计

**生命周期（`service.py` `_process_collector`，已核实）：**

- 每源进入时：`CollectorRun(collector_name, batch_id, trigger_type, start_time, status='running')` → `db.add()` + `db.commit()`（独立线程、独立 Session）。
- 每源完成时：更新 `status`(success/partial/failed)、`fetched_raw`/`created`/`analyzed`、`end_time` → `db.commit()`。

**实时性结论（关键利好）：**

- **并发下各源状态对其他 HTTP 请求实时可读。** 每个源在独立线程用独立 Session，每完成一个就 commit 一次；PostgreSQL 默认 READ COMMITTED 隔离级别下，其他事务（即轮询的 HTTP 请求）可**立即读到已 commit 的行**。
- 因此「已完成数 / 运行中数 / 成功数 / 部分数 / 失败数」无需任何后端改造，**轮询 `collection-logs?status=running` 就能真实看到变化**。
- 「当前正在处理的数据源」= `collection-logs/{batch_key}/runs` 中 `status='running'` 的行，实时准确。

**注意点：**
- 并发 `as_completed` 顺序不确定 → 逐源「顺序进度」无意义，UI 应展示**聚合计数**（success/partial/failed/running 计数），而非逐源线性进度。
- 若某源线程异常崩溃（非被 except 捕获的软失败），该源 CollectorRun 可能停留在 `running` → 产生 **running 孤儿**（见 §七.5、§十三）。

## 四、当前 API 能力审计

**现有两接口（`admin_data_sources.py`）已完全覆盖实时进度需求：**

| 接口 | 能力 | 实时进度可用性 |
|------|------|----------------|
| `GET /api/admin/data-sources/collection-logs` | 按 `COALESCE(batch_id, start_time::text)` 聚合批次；支持 `status=running` 筛选；返回 `started_at/finished_at/source_count/success_count/partial_count/failed_count/running_count/trigger_type/batch_id/duration_seconds` | ✅ 直接用于「当前批次总览」 |
| `GET /api/admin/data-sources/collection-logs/{batch_key}/runs` | 某批次逐源明细（`_run_to_dict`） | ✅ 直接用于「当前正在处理的数据源 / 逐源状态」 |
| `GET /api/tasks/{task_id}` | 后台任务状态（status/progress/step/result/error） | ✅ 用于「任务是否结束」与 step 文案；但**缺 batch_id** |
| `POST /api/collector/run` | 触发采集，返回 `task_id` | ✅ 触发入口 |
| `GET /api/collector/status` | 内存采集状态（last_run/total_collected） | ⚠️ 重启丢失，不用于进度 |

**结论：不必须新增 API。** MVP 阶段复用上述接口即可。若做 task_id↔batch_id 关联，仅**扩展** `/tasks/{task_id}` 的返回值（加 `batch_id` 字段），属非破坏性扩展。

## 五、实时通信方案比较（必须明确推荐）

| 方案 | 实现成本 | 实时性 | 与现有基建契合 | 结论 |
|------|----------|--------|----------------|------|
| **A. 前端定时轮询** | 低。复用 `api/index.ts` 的 `pollTask` + 新增对 `collection-logs?status=running` 的 2s 轮询 | 2s 粒度（采集每源耗时秒级，足够） | ✅ 极高（pollTask 已存在） | **✅ 推荐** |
| B. SSE（Server-Sent Events） | 中。需新增 `/collector/stream/{task_id}` 端点、维护长连接、心跳、断线重连 | 亚秒级 | ❌ 需新协议/端点 | 过度设计 |
| C. WebSocket | 高。需 ASGI/WS 通道、连接管理、鉴权、广播 | 亚秒级 | ❌ 当前 FastAPI+uvicorn 同步为主，改动大 | 过度设计 |

**推荐：方案 A（轮询）。** 理由：
1. 采集是**慢操作**（每源秒级、整批数十秒~分钟），2s 轮询完全满足「实时观感」，无需亚秒级推送。
2. 前端**已有 `pollTask` 轮询基建**（`intervalMs` 默认 1500），直接复用，零新协议。
3. 后端**零新端点**（仅可选扩展 `/tasks/{id}` 返回 batch_id）。
4. 轮询天然容错：断网/后端重启后下次轮询自愈；SSE/WS 需额外重连逻辑。
5. 不存在「多用户并发编辑同一资源」场景，无需服务端主动推送。

**轮询策略建议：**
- 任务生命周期轮询：`GET /tasks/{task_id}`，复用 `pollTask`（1.5s，超时 10min）。
- 批次进度轮询：`GET collection-logs?status=running`（2s），取 started_at 最新的 running 批次。
- 当 `/tasks/{task_id}` 返回终态 **或** running 批次消失 → 停止两路轮询，刷新历史表。

## 六、实时进度 UI 方案审计（三方案，必须推荐一个）

**当前 `CollectionLog.vue` 结构（已核实）：** toolbar（筛选+刷新按钮）+ `.card`（12 列表格，已修 `overflow-x:auto`）+ 分页器；`onMounted→reload()` 一次性加载；**无定时器、无实时刷新、无进度区**。

**方案对比：**

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **方案 1：顶部「当前采集任务」独立区** | 表格上方新增一个卡片区，展示：批次、已运行时间、数据源总数、已完成/运行中/成功/部分/失败计数、当前正在处理的数据源；完成后收起/变「本次采集已完成」提示 | 与历史日志语义清晰分离；生命周期明确（有 running 才显示）；不污染历史表格 | 需新增区块+轮询逻辑 |
| 方案 2：running 批次置顶表格 | 在现有表格顶部插入一行/横幅展示 running 批次（与历史行同结构） | 改动小，复用表格行样式 | 与「未完成的历史批次（running 孤儿/早期无 batch_id）」视觉混淆；终态后需从表格移除，闪动 |
| 方案 3：独立任务状态弹层/抽屉 | 点击「开始采集」后弹出抽屉持续展示进度 | 不占页面空间 | 与「采集日志」页面割裂；用户需停留在弹层；与「在采集日志页面看到进度」需求不符 |

**推荐：方案 1（顶部「当前采集任务」独立区）。** 理由：
- 最贴合需求原话「在『数据管理 → 采集日志』页面实时看到当前采集进度」。
- 与历史批次（含 running 孤儿、早期无 batch_id 的「历史」批次）**视觉与语义隔离**，避免误读。
- 生命周期自然：检测到 running 批次→显示；批次结束→收起并刷新历史表，无残留脏数据。
- 可与现有 toolbar/table 结构共存，不需重构表格。

## 七、必须识别的边界问题（10 项 + 2 项补充）

1. **task_id↔batch_id 无关联竞态**（§一.3）：手动与定时重叠时，纯前端「取最新 running 批次」会张冠李戴。→ 建议建立关联（§八.5）。
2. **5 秒防抖 `CollectorThrottled`**：政府网站采集距上次 < 5s → 抛异常，任务直接 `failed`，error 含「collector running too frequently」。前端须识别 failed+该 error，提示「操作过于频繁，请稍后再试」，而非当成采集失败。
3. **定时 cron 与手动重叠**：定时任务无 task_id，可能与手动并发产生两个 running 批次；进度面板须能呈现「可能不止一个 running 批次」（或仅展示用户 task_id 关联的那个）。
4. **task 内存态重启丢失**：后端重启后 `_tasks` 清空，`GET /tasks/{task_id}` 返回 None → 前端轮询应超时降级到「按 running 批次轮询」（running 状态仍在 DB，不丢）。
5. **running 孤儿**：异常中断/线程崩溃导致 CollectorRun.status='running' 永久残留 → `collection-logs?status=running` 永远返回该孤儿，面板永不结束。→ 需孤儿超时收尾（建议：started_at 超过阈值如 30min 仍 running 视为终态），本期**暂不处理**，列入 §十三。
6. **on_progress 的 name 是「刚完成源」非「正在处理源」**：UI 若直接把 step 文案当「正在处理」会误导。→ 用 `status='running'` 的 CollectorRun 判断真实在处理的源（§三）。
7. **并发完成顺序 ≠ 进度顺序**：`as_completed` 顺序不确定，done 计数正确但逐源顺序乱；UI 用聚合计数（running/success/partial/failed），不用逐源线性顺序。
8. **时区**：`start_time` 入 `timestamp without time zone` 被存成本地墙钟，前端 `new Date()` 解析无 8h 偏移 → 「已运行时间 = now - started_at」计算正确。当前无 bug，但若后续改 UTC 存储需同步校验（记录备查）。
9. **聚合阶段展示**：采集完成后仍跑 `auto_aggregate_after_collect`，此阶段 CollectorRun 全终态但 task 仍 running → 面板应展示「采集完成，正在聚合事件…」作为终态前最后阶段，避免误判失败。
10. **前端轮询泄漏**：CollectionLog.vue 卸载时若未 `clearInterval`/`clearTimeout` 会泄漏定时器；多个标签页同时打开会各起轮询加重后端。→ `onUnmounted` 清理 + 可选 `document.hidden` 时暂停轮询。
11. **（补充）权限/401**：`collection-logs` 需 `require_admin`；非 admin 调会 401 → 触发已知「token 401 静默空屏」问题（不在本次范围），轮询失败须有降级而非卡死。
12. **（补充）无取消接口**：用户可能想「停止采集」，当前无取消端点 → 本期面板只展示不提供停止（如需，列入后续）。

## 八、完整审计报告（13 部分）

### 8.1 当前系统已经具备的实时能力
- CollectorRun 并发各源独立 commit，其他 HTTP **实时可读**（§三）。
- `collection-logs?status=running` 已返回实时聚合统计：source_count / success_count / partial_count / failed_count / running_count / started_at / batch_id / trigger_type。
- `collection-logs/{batch_key}/runs` 已返回逐源实时状态（含 status='running' 的正在处理源）。
- 前端已有 `pollTask` 轮询基建（`api/index.ts`）。
- 后端 `Task` 已有 progress/step 内存进度（虽信息薄）。

### 8.2 当前系统缺少的能力
- task_id 与 batch_id **无关联**（task 对象/result 无 batch_id）。
- 前端**无**「当前采集进度」展示区、**无**轮询触发逻辑、**无**「进行中任务」跨组件传递机制（AppLayout 拿到 task_id 后未告知 CollectionLog）。
- 「当前正在处理的数据源」**无准确来源**（on_progress 的 name 是刚完成源）。
- 进度面板**无**生命周期管理（开始/结束/收起/刷新历史表）。
- 定时器清理、降级、401 处理等健壮性逻辑缺失。

### 8.3 最小实现方案
- **前端为主，后端可选小改**。
- 前端：在 CollectionLog.vue 顶部加「当前采集任务」区；通过 pinia store 或事件拿到进行中 `task_id`；轮询 `/tasks/{task_id}`（复用 pollTask）+ `collection-logs?status=running`；渲染聚合计数与 running 源；终态后停轮询并 reload 历史表。
- 后端（可选但推荐）：建立 task_id↔batch_id 关联（§八.5），使面板精确对应「用户点的那次」。

### 8.4 是否新增 API
- **否（MVP 不新增）。** 复用 `collection-logs`、`collection-logs/{batch_key}/runs`、`/tasks/{task_id}`、`/collector/run`。
- 若做 task_id↔batch_id 关联，仅**扩展** `/tasks/{task_id}` 返回值（加 `batch_id`），非破坏性，不新增端点。

### 8.5 是否需要 task_id 与 batch_id 关联
- **建议做**，以根除手动/定时重叠竞态（§一.3、§七.1）。
- **最小关联方案（三处小改）：**
  1. `service.py`：`collect_and_analyze_concurrent` 在 `return merged` 前把 `batch_id` 带上（在 `CollectorRunResult` 加 `batch_id` 字段，或在 `_run_collect_task` 内捕获）。注意：`batch_id` 在函数内生成，可直接在函数返回处附上。
  2. `collector.py` `_run_collect_task`：拿到 batch_id 后写 `task.batch_id = batch_id`（在 on_progress 首次触发或函数返回前均可；建议在 `collect_and_analyze_concurrent` 返回后从 result 取 batch_id 写入 task）。
  3. `task_manager.py`：`Task` 增加 `batch_id: Optional[str] = None` 字段，`to_dict()` 输出 `batch_id`。
- **零后端改动降级方案（若坚持不改后端）：** 前端轮询 `collection-logs?status=running`，取 `started_at` 最新的 running 批次作为「当前」。适用于「通常单次采集」场景；代价是重叠时可能展示错误批次（已知妥协）。

### 8.6 通信方案选择
- **轮询（方案 A）**，见 §五。复用 `pollTask` + `collection-logs?status=running`，不引入 SSE/WebSocket。

### 8.7 前端 UI 结构（推荐方案 1）
```
CollectionLog.vue
├─ [新增] 当前采集任务区（仅在检测到 running 批次 / 进行中 task 时显示）
│   ├─ 批次号（batch_id 前 8 位） + 触发方式（手动/定时）
│   ├─ 已运行时间（now - started_at，实时刷新）
│   ├─ 数据源总数 / 已完成(成功+部分) / 运行中(running_count)
│   ├─ 成功 / 部分 / 失败 计数（彩色标签）
│   ├─ 当前正在处理：status='running' 的 CollectorRun.source_name 列表
│   └─ 任务 step 文案（来自 /tasks/{task_id}.step，如「正在自动聚合事件…」）
├─ toolbar（筛选 + 刷新，保留）
├─ .card（历史批次表格，保留，overflow-x:auto）
└─ pager（保留）
```
- 进行中 task 来源：`AppLayout.handleCollect` 触发后，将 `task_id` 写入 pinia store（如 `useCollectStore().activeTaskId = task_id`）；CollectionLog 挂载/激活时读取，若不存在则按 running 批次轮询。

### 8.8 完整数据流（从点击到完成）
1. 用户点「开始采集」→ `AppLayout.handleCollect()` → `POST /api/collector/run` → 得 `{task_id}`。
2. `handleCollect` 将 `task_id` 写入 store（`activeTaskId`），并 `pollTask(task_id)`（既有逻辑，弹消息+data-refresh）。
3. CollectionLog.vue 检测到 `activeTaskId` 存在（或自身轮询发现 running 批次）→ 显示「当前采集任务」区。
4. 前端两路轮询：
   - A：`GET /tasks/{task_id}`（1.5s）→ 取 status / step /（关联后）batch_id。
   - B：`GET collection-logs?status=running`（2s）→ 取最新 running 批次聚合统计 + 其 `{batch_key}/runs` 中 `status='running'` 源。
5. 面板实时渲染：已运行时间、总数、完成/运行中/成功/部分/失败、正在处理源。
6. 终止条件：`/tasks/{task_id}` 返回 success/failed **或** running 批次消失 → 停止两路轮询，隐藏「当前采集任务」区，reload 历史表（等价于 data-refresh）。
7. 若 failed：区分 `CollectorThrottled`（提示频繁）与其他错误（提示失败原因），不展示为「进度完成」。

### 8.9 异常与并发边界
- 见 §七 的 12 项（竞态、5s 防抖、定时重叠、重启丢失、running 孤儿、name 语义、完成顺序、时区、聚合阶段、轮询泄漏、401、无取消）。
- 核心兜底：轮询失败/超时 → 降级到 running 批次轮询；running 批次消失即视为结束；孤儿批次（§十三）本期不自动收尾。

### 8.12 验证方案
- **功能验证：** 点「开始采集」→ 顶部进度区应立即出现；观察「已运行时间」递增、running_count 递减、success/partial/failed 递增；展开「当前正在处理」应显示真实 running 源名。
- **数据一致性：** 进度区计数应等于 DB `SELECT status, count(*) FROM collector_runs WHERE batch_id=:b GROUP BY status`（直接查生产库核验）。
- **终态验证：** 采集完成 → 进度区收起、历史表出现新批次、data-refresh 触发。
- **边界验证：**
  - 5s 内连点两次 → 第二次 failed 且提示「操作过于频繁」，不卡死。
  - 采集进行中刷新页面 → 进度区应基于 running 批次重新出现（不依赖内存 task）。
  - 模拟后端重启（task 丢失）→ 轮询降级到 running 批次，面板仍正常。
  - 多标签页 → 仅活动标签轮询（若实现 visibilitychange）；无定时器泄漏（切走再回无叠加）。
  - 非 admin token → 401 有降级提示而非空白。

### 8.13 风险与暂不处理事项
- **【附带发现-S2 渲染缺陷】**：部署产物 `index-DEGzqvaR.js` 中「历史」span 为**静态无条件渲染节点**（缺 `v-else`），手动/定时批次会多渲染一个「历史」药丸。此为前序 S2 补丁遗留缺陷，**非本次范围**，建议另立最小修复（补回 `v-else` 或重建）。审计阶段不改。
- **running 孤儿清理**：异常中断产生的永久 running 批次本期不自动收尾；建议后续加「started_at 超时阈值」定时巡检或将状态置终态。本期进度面板在孤儿存在时会一直显示，需在验证时人工规避或临时清理。
- **取消采集**：本期不提供「停止采集」按钮（无取消端点）。
- **多 running 批次重叠展示**：若不做 task_id↔batch_id 关联，重叠时面板可能展示非用户触发的批次（已知妥协）。
- **时区**：当前本地墙钟存储无 bug，记录备查；若后续迁移 UTC 需同步校验「已运行时间」计算。
- **401 静默空屏**：已知前端全局问题，不在本次范围；轮询失败建议加降级。
- **后端重启丢失**：task 内存态特性，进度面板以 DB running 批次为准可缓解，已写入验证方案。
