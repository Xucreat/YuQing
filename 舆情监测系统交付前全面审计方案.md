## 0. 审计总纲

### 0.1 目标

判定系统是否达到企业项目交付标准，产出：交付成熟度评分（0–100）、P0–P3 问题分级清单、必须修复 / 可延期 / 不建议继续投入三张清单。

### 0.2 审计铁律（全程强制）

| # | 约束 | 执行手段 |
|---|------|---------|
| 1 | **只读审计，零修改** | 全程仅使用 SELECT 查询、文件读取、GET 类接口调用；不执行任何 UPDATE/DELETE/INSERT/DDL、不改代码、不改配置、不重启服务 |
| 2 | 不做功能扩展 | 发现缺失功能只记录为问题项，不实现 |
| 3 | 不降低标准 | 测试失败/覆盖不足如实记录，禁止改测试、改断言、跳过用例 |
| 4 | 一切结论必须有证据 | 每个问题项须附：代码文件+行号 / SQL 查询结果 / 接口实测响应 / 日志片段，四选一以上 |
| 5 | **数据库身份门禁** | 任何连库操作前先跑 `scripts/db_identity_check.py`（退出码 0 = VERIFIED 真实库）；真实生产库为 `Desktop\舆情监测系统\pgdata`，严禁误连早期空克隆库 |
| 6 | 生产服务不中断 | 审计期间不杀 uvicorn 进程、不做压测到打挂服务的程度（性能验证限制并发与时长） |

### 0.3 审计方法

- **静态审查**：代码走读（router/service/model 分层）、grep 扫描（危险模式、硬编码密钥、TODO/FIXME）、依赖清单核对。
  - ⚠️ 注意本机 node 虚拟化文件系统坑：node 写过的源码文件必须经 node 读取（`node -e "readFileSync"`）才是明文，直接 cat/Read 可能读到压缩字节，防止把"乱码"误判为代码损坏。
- **动态验证**：只读 API 实测（curl，含带 token / 不带 token / 低权限 token 三种身份）、只读 SQL 核查、日志抽样。
- **数据核对**：以生产库真实数据验证一致性（行数、孤儿记录、重复率、索引使用）。
- **测试执行**：在测试库 `:5432/opinion_test`（`DB_IDENTITY_CHECK=off`）运行现有 pytest 套件，只记录结果不修改用例。
- **限制声明**：本机无 Chromium/Playwright，前端不能做自动化截图/E2E；前端审计以代码审查 + 构建验证 + API 层联调核对代替，此为已知审计盲区，将在报告中如实标注。

### 0.4 已知事实基线（来自项目历史记录，审计中逐项复核而非默认采信）

以下为历史会话已确认的疑点/事实，作为审计的**优先验证清单**：

1. 前端 axios 无全局 401 拦截器 → token 过期静默空屏（已知未修复）。
2. `alert_records` 存在历史数据：74 条 critical / 9 条 low 为旧模型产物，未回灌重算。
3. `opinions.risk_score` 历史数据未按 risk-v2.0 重算，新旧口径共存。
4. 采集批次无 `batch_id` 列，靠 `start_time` 隐式聚合。
5. vite 构建产物旧哈希累积（`emptyOutDir:false` + `_d.py` 只写不删）。
6. 曾发生多会话并行导致"新模型+旧库"生产故障（迁移窗口期风险）。
7. PDF 导出 `Content-Disposition` 文件名仅 ASCII。
8. conftest 默认指向不可达的 `:5433`，测试须手动覆盖 `DATABASE_URL`。

## 二、核心业务流程验收

**方法**：每条链路按「配置 → 触发 → 中间态 → 落库 → 展示 → 异常分支」做端到端追踪，代码走读 + 生产库只读 SQL 验证 + 只读 API 实测。

### 2.1 数据采集流程

| 环节 | 验收要点 | 验证方式 |
|------|---------|---------|
| 数据源配置 | CRUD 完整性、启停开关生效、装配可靠性（引用《数据源配置模型与采集器装配可靠性优化_实施报告》复核落地） | 走读 `admin_data_sources.py` + `data_source.py`；SQL 核对 22 个监测信源状态 |
| 采集任务创建/调度 | 手动 `/collector/run`（并发）与 cron 定时（顺序）双路径是否共用幂等保护；scheduler 单实例保证 | 走读 scheduler `_run_collector_job`、`collect_and_analyze[_concurrent]` |
| 数据入库 | 幂等性（复核采集幂等性加固 Phase7 成果）：重复 URL/内容去重键、并发写冲突处理 | SQL：按去重键统计 opinions 重复率 |
| 采集日志 | `collector_runs` 记录完整性：status 终态覆盖率、start/end_time 空值率、按 start_time 批次聚合可用性（无 batch_id 的隐患评估） | SQL 聚合分析近 7 天 runs |
| 异常处理 | 单源失败是否拖垮整批；失败 run 是否留痕；网络超时/解析失败的降级路径 | 代码走读 + 检索失败 run 样本 |

### 2.2 舆情分析流程

- **清洗/去重**：入库前清洗规则位置与覆盖面；标题格式化（`test_title_format` 对应逻辑）；发布时间解析回填（`backfill_pubtime.log` 遗留任务是否收口）。
- **聚合/事件生成**：`event/aggregator.py` 聚合口径；Phase7.5 幂等收口（advisory lock）是否覆盖并发聚合；`event_opinions` 关联完整性（孤儿关联 SQL 核查）；历史重复事件是否已按 Phase7.5-B 清理干净。
- **情感分析**：AI 服务失败时的降级值；sentiment 空值率。
- **风险评分**：risk-v2.0 链路验证——`risk_factors` JSONB 落库率、`risk_model_version` 分布（新旧版本共存比例）、`risk_category` 派生正确性抽样、正面误报保护逻辑抽样（positive+无危害词是否确实无 high）。
- **验证 SQL 示例**（只读）：
  ```sql
  SELECT risk_model_version, count(*) FROM opinions GROUP BY 1;
  SELECT count(*) FROM opinions WHERE sentiment='positive' AND risk_score>=70;  -- 应仅危害词命中样本
  SELECT count(*) FROM event_opinions eo LEFT JOIN opinions o ON o.id=eo.opinion_id WHERE o.id IS NULL;
  ```

### 2.3 预警流程

- 风险触发：`alert_service.evaluate` 的 (rule, opinion) 去重机制在规则修改后的行为。
- 预警生成：`risk_level` 由 `_map_risk_level(risk_score)` 派生（Phase1 改造）的落地验证；trigger_reason 完整性。
- 状态流转：5 态（pending/processing/resolved/ignored/false_positive）CHECK 约束生效验证；`handled` 双写一致性 SQL 核查（status ∈ 终态 ⇔ handled=true）。
- 处置闭环：handle API 权限（alerts:write）、审计留痕（HANDLE_ALERT 写入 `user_operation_logs`）、无 body 旧调用兼容。
- **历史数据风险**：74 critical/9 low 旧口径记录对交付演示与统计报表的污染程度评估（只评估，不重算）。

### 2.4 指挥大屏流程

- 数据口径：`/api/dashboard/*`（hot-keywords、region-children、risk-distribution、alert-stats）实测返回与库内 SQL 独立复算对比（total≈429、high_risk、监测信源 22 等口径复核）。
- 实时刷新：轮询间隔与后端查询成本（EXPLAIN 只读分析大屏 SQL）。
- 页面稳定性：自动轮播 tour 的内存泄漏风险（定时器清理）代码审查；token 过期后大屏静默失效问题（已知 401 无拦截）在无人值守大屏场景的严重度**升级评估**——大屏是长时间挂机场景，此问题可能是 P0/P1 级。

### 2.5 输出

每条链路给出「闭环 / 半闭环 / 断裂」判定 + 缺失环节清单。

## 三、安全审计

### 3.1 认证

| 项 | 方法 |
|----|------|
| 登录流程 | 走读 `auth.py`：失败次数限制/锁定？登录日志落库（LoginLogs 页面对应表）？ |
| Token 管理 | JWT 算法与密钥强度（是否硬编码/弱密钥）、过期时间、有无刷新机制、注销后 token 是否仍有效（无黑名单=记录风险） |
| 密码安全 | 哈希算法（bcrypt/argon2 vs md5/sha1）、密码复杂度策略、初始密码/默认账号（SQL 查 users 表弱账号）、修改密码是否使旧 token 失效 |

### 3.2 授权（RBAC 真实性验证）

- 用低权限角色 token 实测调用管理接口（`/admin/data-sources`、users、roles、collector/run），逐个验证 403；**重点扫描全部 15 个路由模块中未挂权限依赖的端点**（生成"裸奔接口"清单）。
- 越权路径：横向越权（普通用户读他人数据）、参数越权（handle 别人处置中的告警）。
- `require_admin`、`alerts:write` 等权限点与 permissions 表数据一致性核对。

### 3.3 审计日志

- 登录日志、操作日志（表名 `user_operation_logs`）覆盖面：哪些高危操作**未**留痕（删除数据源？改角色权限？改关键词权重？）→ 生成留痕缺口清单。
- 日志本身的防篡改/防删除评估。

### 3.4 其他安全面

- SQL 注入（是否全 ORM 参数化，grep 裸 `text()`/f-string 拼 SQL）；CORS 配置；敏感信息泄露（错误响应是否带堆栈）；`collector_router` 无认证暴露面评估；`.env`/密钥是否进 git 历史。

### 3.5 输出

安全等级评价（高/中/低危分级）+ 高风险问题列表（每项含复现步骤）。

## 四、数据库审计

**前置**：`db_identity_check.py` 通过后，仅以只读连接执行。

| 检查项 | 方法（全部只读 SQL） |
|--------|---------------------|
| 表结构合理性 | 全表 `\d+` 审查：字段类型、nullable 合理性、缺失的 NOT NULL/DEFAULT；19 个迁移与实际 schema 一致性（`alembic current` == p11_phase2b2 且 `alembic check` 无漂移） |
| 索引情况 | `pg_indexes` 全量 vs 高频查询路径（opinions 按 publish_time/risk_score/created_at、alert_records 按 status、collector_runs 按 start_time、event_opinions 联合键）；`pg_stat_user_indexes` 找无用索引；缺失索引用 EXPLAIN 验证 |
| 数据一致性 | 孤儿记录（event_opinions→opinions/events、alert_records→opinions/rules、handled_by→users）；alert `handled` 双写一致性；events 状态与聚合口径一致性 |
| 外键/关联 | 有多少关联是"逻辑外键"（无 FK 约束）→ 评估数据完整性风险 |
| 历史数据问题 | 旧风险口径记录量化（risk_model_version IS NULL / 旧版本占比）、74 critical 历史告警、publish_time 回填完成度 |
| 重复数据风险 | opinions 按 (url) / (title+source) 重复率；events 重复（Phase7.5-B 清理后残留复查） |
| 增长后性能 | 当前各表行数与日增速估算 → 3/6/12 个月投影；大屏聚合 SQL 在 10 倍数据量下的 EXPLAIN 成本预估；无分区/归档策略的影响评估 |
| 备份有效性 | `backup/`、`backups/` 中 dump 的时效性（最新一次时间）、是否有恢复演练记录 → 归入第八章交叉验证 |

## 五、API 质量审计

**方法**：从 FastAPI app 提取全量路由表（openapi.json，只读），逐项打分。

- **命名规范**：REST 风格一致性；`/api` vs `/admin` vs `/collector` 三前缀的规范性。
- **参数校验**：Pydantic schema 覆盖率；分页参数上限（page_size 无上限 = 拖库风险）；日期/枚举参数非法值实测（只读接口上做边界测试）。
- **返回格式一致性**：是否统一响应包裹；错误响应结构一致性（422/400/401/403/404/500 各类实测一次）。
- **错误处理**：全局异常处理器有无；500 是否泄露堆栈/SQL。
- **权限校验完整性**：与第三章 3.2 共用"裸奔接口"扫描结果。
- **未使用接口**：路由表 vs 前端 `api/` 目录调用清单交叉比对（用 node 读前端源码防乱码）→ 死接口清单。
- **危险接口**：无鉴权的写操作、批量删除、`/collector/run` 可否被外部反复触发（DoS/脏数据）、debug 类端点（对照根目录 `add_debug.py` 检查是否有调试残留挂到了 app 上）。

| 单机风险 | 无高可用/单点（应用+DB 同机）——按交付形态（内网单机交付 vs 服务化交付）分别给出结论 |

## 九、交付风险评级（最终输出物定义)

### 9.1 交付成熟度评分模型（0–100）

| 维度 | 权重 | 评分依据 |
|------|------|---------|
| 核心业务链路闭环 | 25 | 第二章四条链路判定 |
| 安全 | 20 | 第三章高危问题数量与等级 |
| 数据质量与数据库 | 15 | 第四章一致性/性能结论 |
| 架构与代码质量 | 10 | 第一、五章 |
| 前端质量 | 10 | 第六章 |
| 测试体系 | 10 | 第七章通过率与覆盖 |
| 部署运维成熟度 | 10 | 第八章 |

评分规则：每维度 0–100 按权重加总；**存在任一 P0 时总分封顶 69**（不允许"高分但阻塞"的误导性结论）。

### 9.2 问题分级标准

- **P0 阻塞交付**：数据错误/丢失、权限绕过、核心链路断裂、生产无法稳定启停。
- **P1 建议交付前修复**：已知会在正常使用中触发的缺陷（如 401 静默空屏、历史告警口径污染统计）、高危操作无审计。
- **P2 后续优化**：工程债（conftest 指向、构建产物累积、命名不一致）、覆盖率缺口。
- **P3 长期优化**：架构演进（batch_id、分区归档、高可用、E2E 测试体系）。

### 9.3 最终交付物

执行完成后输出《舆情监测系统交付前全面审计报告》，包含：

1. 交付成熟度总分 + 七维度雷达分。
2. 全量问题清单（每项：编号 / 等级 / 证据 / 影响 / 修复建议 / 预估工作量）。
3. **必须修复列表**（P0 + 高影响 P1）。
4. **可以延期列表**（P1 低影响 + P2）。
5. **不建议继续投入列表**（投入产出比差的项，如当前形态下的浏览器全兼容、多机高可用）。
6. 审计盲区声明（无头浏览器缺失导致的前端验证局限等）。

### 9.4 执行顺序与依赖

```
D1  库身份门禁确认 → 四(数据库) + 一(架构静态审查)   [互不依赖，可并行]
D2  二(业务链路) —— 依赖 D1 的库结论
D3  三(安全) + 五(API) —— 共用路由表与多身份实测
D4  六(前端) + 七(测试执行)
D5  八(部署) → 九(汇总评级出报告)
```

每日审计发现即时记录，不等到最后汇总；任何疑似 P0 一经发现立即单独上报，不压到报告发布日。

*本方案为只读审计计划。在获得批准前，不执行任何修改类操作；审计过程中所有数据库访问均为 SELECT，所有接口调用均为幂等只读或在测试库进行。*
