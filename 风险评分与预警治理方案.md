# 风险评分与预警历史数据治理方案（Phase）

> 状态：**方案设计** —— 本文档仅描述治理设计，未执行任何代码或数据修改。
> 生成时间：2026-07-24 15:02
> 约束遵循：不删除历史数据 / 不修改采集链路 / 不引入 AI 模型 / 不扩大功能范围。

## 一、当前问题

预警中心 `alert_records` 共 **79 条 = 70 critical + 9 low**。

- **70 条 critical 中 69 条 `severity_score = 0`**（即 Phase 2-A 风险重算前采集、从未过重算的历史数据）；仅 1 条（7/24 13:00 真实溃坝事故，`severity_score=90`）判定正确。
- **70 条 critical 中 41 条情感为 `positive`（正面舆情）**，典型为政策解读、会议报道、便民服务新闻，被误判为"高风险安全舆情"。
- 二次精确核对：41 条正面 critical 中 **仅 15 条真正命中真实危害词**（事故/火灾/死亡等），其余 26 条由语境词（群体 22 条、维权 2 条等）或规则阈值误升触发。
- Phase 2-A 部署后（7/24 12:35）新采集数据**基本干净**：仅新增 1 条 high/critical，且 1007 条 opinions 中仅 2 条 `severity_score≥70`（真实事故，判定正确）。

**结论**：误判集中在**历史 critical 记录**，新数据管线已正确；问题本质是"历史数据未按现行评分标准重算"。

## 二、根因

### 2.1 审计确认（只读）
| 审计项 | 结论 |
|---|---|
| `alert_records` 生成逻辑（`alert_service.py:77-143`） | 当前代码 `derived_level = _map_risk_level(opinion.risk_score)`，**不再读 `rule.risk_level`**；`rule_level` 覆盖问题**仅存在于 Phase 1 前的历史数据** |
| `opinions` 风险字段 | `severity_score=0` 的历史记录从未经过 `RiskEngine.refine()`，其 `risk_score` 仍是旧 `RuleFallbackProvider` 口径（受已治理的"群体/维权"weight 影响） |
| `RiskEngine` 调用链（`collectors/service.py:343-395`） | 仅在**新采集**时调用并写回；无历史回填入口 |
| `AlertService.evaluate` 流程 | 按 `(rule_id, opinion_id)` 去重（`:66-75`），**已存在的 alert 永不重算** |

### 2.2 误判四因（历史数据视角）
1. **等级曾直接抄规则配置**：59 条集中于 7/23 17:00 批量生成（Phase 1 前），代码为 `risk_level = rule.risk_level`，把 Rule 5 的 `critical` 原样抄给每条匹配记录。
2. **语境词当敏感词**：早期"群体" sensitive weight=7 → `risk_score=20+70=90≥80` 命中 Rule 5（threshold=80）。Phase 1 已置 0，但旧记录未重算。
3. **纯子串匹配无语境**："死亡率下降"命中"死亡"(sev=90)、"防范事故工作会议"命中"事故"。当前 `word in text` 无法区分发生/防范。
4. **历史记录无重算机制**：`evaluate()` 去重跳过 → 历史 critical 永不自动修正。

## 四、数据影响范围

### A. 历史 `opinions` 重新评分策略
- **目标集**：`severity_score = 0 AND risk_model_version IS NULL`（Phase 2-A 前/未重算数据；实测绝大多数历史 opinions 命中）。
- **方法**：逐条用 `RiskEngine(severity_keywords=get_severity_keywords(db)).refine(title, content, opinion.sentiment)` 重算。
  - 复用 `opinion.sentiment`（采集时已产出，**不重跑** `RuleFallbackProvider`，故不触碰 AI/采集链路）。
  - 写回：`risk_score / severity_score / event_state / resolution_flag / risk_factors / risk_model_version("risk-v2.0") / risk_category`。
  - **不写**：`analysis_status / keywords / summary / ai_*` 等既有字段。
- **预期**：原"群体"weight 已为 0，重算后大量正面新闻 `risk_score` 显著下降、`severity_score` 维持 0（无真实危害词）。

### B. `alert_records` 重新生成/更新策略（就地 UPDATE，非删除重建）
- **关键决策**：**不删除、不重建** alert，以保留 `status / handled / handled_by / handled_at / handle_note / created_at / rule_id` 全部审计字段（满足"不删除历史数据"）。
- **方法**：遍历 `alert_records`，加载其 `opinion`，用与 `evaluate()` 完全一致的派生逻辑重算等级：
  ```
  level = _map_risk_level(opinion.risk_score)          # ≥70→high / ≥40→medium / else→low
  if opinion.severity_score >= 70: level = "critical"
  if opinion.sentiment == "positive" and level in (high, critical):
      if 未命中 HARM_INDICATOR_KEYWORDS: level = "low"
  ```
  仅 **UPDATE `risk_level` 与 `trigger_reason`** 两列；`trigger_reason` 前缀加 `[recomputed @ <ts>]` 便于追溯。
- **预期**：70 critical → 多数降为 low/medium；命中真实危害词者保留 critical/high；9 low 基本不变。

### C. 审计字段保留方案
- 所有 `status / handled* / handle_note / created_at / rule_id / rule_name / opinion_id` **原样保留**，仅改 `risk_level + trigger_reason`。
- 在 `user_operation_logs`（即 `OperationLog` 表，与现有 `audit_write` 一致）写 1 条治理审计记录；不新增任何表/列。
- 不修改表结构，**无需 Alembic 迁移**（目标列 `severity_score / risk_factors / risk_model_version / risk_category / alert_records.risk_level / trigger_reason` 均已由 Phase 2-A/B 迁移就绪）。

### D. 是否需要数据库迁移
- **否**。本方案为纯数据回填（UPDATE），不涉及任何 DDL/列变更。所有目标列已存在。

## 五、回滚方案

因无 DDL 变更，回滚=用备份数据 UPDATE 还原，无需 migration downgrade。

1. **前置备份（强制）**：实施前执行 `pg_dump` 导出 `opinions` + `alert_records` 整表（或全库）至 `backups/`。
2. **变更前快照**：脚本先写出 CSV 快照
   - `opinions`: `id, old_risk_score, old_severity_score, old_risk_factors, old_risk_model_version, old_risk_category`
   - `alert_records`: `id, old_risk_level, old_trigger_reason`
   存放于 workspace 备份目录，回滚时按 `id` 逐行 UPDATE 还原。
3. **回滚命令**：读取快照 CSV，对每张表执行 `UPDATE ... SET ... WHERE id = :id`（同一事务分批提交）。
4. **验证**：回滚后重新查询 `alert_records` 等级分布，应恢复至 70 critical + 9 low。

## 六、实施步骤（待用户确认后执行）

1. **备份**：`pg_dump` 全量 + 导出 CSV 快照（强制前置，未备份不执行后续）。

4. **用户确认**：呈现 dry-run 报告，由用户拍板是否落地。
5. **真实执行**：事务内分批 `UPDATE`（每批 commit，失败可定位断点）；先 opinions 后 alerts（alert 重算依赖新 `severity_score`）。
6. **校验**：重查 `alert_records` 等级分布与 `opinions.severity_score` 分布；抽查样本确认正面新闻已降级、真实事故仍 critical。
7. **审计落库**：写 1 条 `user_operation_logs` 治理记录（含影响行数、执行时间）。
8. **观察**：下一轮采集 + `evaluate()` 后确认无新误判、旧记录未再被错误刷新。
