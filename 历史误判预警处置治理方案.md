## 二、处置方案设计

### 2.1 哪些记录建议标记误报（false_positive）

**建议标记 = 32 条“原 critical → 治理后 low”记录**（即 severity_score=0、无任何真实危害词、纯因旧 `rule_level` 复制逻辑 + 语境词“群体/维权”权重被错误拉高到 critical 的历史告警）。

典型样本（10 条，均 severity=0、sentiment 多为 positive/neutral）：
- 《“努力让每个孩子都能享有公平而有质量的教育”——习近平总书记…》risk=20
- 《把老百姓关切的事一件一件办好》risk=20
- 《重温工运历史 弘扬沂蒙精神》risk=20
- 《切实做好山洪地质灾害防范应对工作》risk=20
- 《总书记为基础教育高质量发展指明方向》risk=20
- 《我市大力实施学校美育浸润行动》risk=20
- 《第一观察丨总书记为基础教育高质量发展指明方向》risk=20
- 《唐山职业技术学院专家走进遵化侯家寨镇 电商技能培训“有料”又“有效”》risk=20
- 《壹时评：别让 APP 沦为贷款营销“围猎场”》risk=20
- 《三部门联合下发通知做好“八一”拥军优属工作》risk=20

> 其中 alert_id=93 已在人工处置中标为 `false_positive`，与建议一致，脚本对其为幂等（无变更）。

### 2.2 哪些记录保留（不标记、不删除）

| 分组 | 数量 | 处置建议 |
|---|---|---|
| critical→critical（真实危害） | 16 | 保留，pending 待人工处置 |
| **critical→medium（可信中等风险）** | 22 | **保留**，不标误报（正文含真实危害词） |
| low→low（原本即低） | 9 | 保留 |
| 32 条 low 中已被人工标 false_positive | 1（alert=93） | 保留人工决策 |
| 22 条 medium 中已被人工处置 | 2（alert=11 resolved / alert=90 false_positive） | 保留人工决策 |

**保留合计 = 79 − 31（自动标记数） = 48 条保持原 status**（其中 16 critical + 22 medium 为“可信风险”核心展示集）。

### 2.3 状态变更影响（status 归位）

对 **31 条 pending 的 critical→low 记录**（自动标记，排除已人工处置的 3 条）：

| 字段 | 变更前 | 变更后 |
|---|---|---|
| `status` | pending | `false_positive` |
| `handled` | False | True |
| `handled_by` | NULL | `system(governance)` |
| `handled_at` | NULL | 执行时间戳 |
| `handle_note` | NULL | `历史误报治理：原critical由旧rule_level复制逻辑误判，重算后severity=0无真实危害词，降级为low` |
| `risk_level` | low | **不变**（不修改 risk_score / 不回改等级） |
| `trigger_reason` | （现有洁净文案） | **不变**（不增 recomputed 标记） |
| `created_at` / 其余审计字段 | — | **全部不变** |

> 性质：仅 status 归位 + 处置闭环字段补全，**非删除、非重建、不触碰 risk_score / RiskEngine / 采集链路**。

### 2.4 是否需要前端过滤优化

**需要（推荐，但非强制）。** 当前 `frontend/src/views/Alerts.vue` 预警记录页：

- 默认进入的是“预警规则”页签；进入“预警记录”页签时，过滤器仅有 `recFilterRisk`（等级）与 `recFilterHandled`（已/未处理），**无 status 过滤、无“排除误报”默认项**。
- 默认 `recFilterHandled=null` → 展示**全部**记录，含 `false_positive`。

为使“默认展示可信风险，同时保留历史审计”落地，有两种等价路径：

- **路径 A（推荐 · 前端小改）**：在“预警记录”页签增加默认开启的「排除误报」开关（`exclude_false_positive`，默认 true）。开启时向 `GET /alerts/records` 传 `status_ne=false_positive`（或后端新增 `exclude_status` 参数），隐藏误报；关闭则显示全部（含历史审计）。不动数据语义，仅需前端 + 后端列表接口支持一个可选参数。
- **路径 B（纯数据 · 不改前端）**：仅做 2.3 的 status 归位。用户仍可在前端用「处理状态=已处理」过滤看到这 31 条（因 handled=True），但默认列表仍会显示它们，需依赖人工切换，体验略弱。

> 前端改动不属于本任务禁止项（禁止项仅：删除 alert_records / 修改 risk_score / 修改 RiskEngine / 修改采集链路）。若采用路径 A，需一次 `vite build` + 重启前端托管；属“展示层优化”，不扩大评分/采集功能范围。

## 三、执行前门禁与一致性（设计，待批准执行时实施）

1. `db_identity_check.py` 通过（退出码 0，确认为生产库 `opinion_db`）。
2. 复用前序 Phase 已生成的 `pg_dump` 全量备份 + `alert_records_snapshot_*.csv`（可精准按 id 回滚 status/handled 字段）。
3. 单事务：31 条 UPDATE + 1 条审计 INSERT 同处一次 commit，避免“status 已改但审计缺失”的中间态。
4. 异常中止阈值（执行时复核）：若待标记数 ≠ 31（例如前序数据被改动）、或 low 记录中出现 severity≥70，立即回滚不提交。

### 3.1 安全性验证（只读确认，消除顾虑）

- `Alerts.vue` 的「执行评估」按钮调用 `POST /alerts/evaluate`，其 `evaluate()` 按 `(rule_id, opinion_id)` 去重、**只新建不更新**已有告警。故：
  - 不会把 31 条 false_positive 回退；
  - 不会因这 32 条舆情现 risk_score=20（低于 Rule 5 阈值 80）而重建 critical 告警。
  → 标记误报后即便误点“执行评估”也安全。
